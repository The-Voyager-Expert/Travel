Section 2 — UX findings for the UX routine
Search input uses type="text" instead of type="search"
City-Transit-Cards.html · <input type="text" id="tc-search">
What was found
The #tc-search input is declared as type="text". The site's shared CSS in web-travel-style.css targets input[type=search] to inject the magnifier icon via var(--search-icon) and style the clear button via ::-webkit-search-cancel-button. Both rules are bypassed entirely on a type="text" input, so the transit cards search box has no magnifier icon and no gold ✕ clear button — inconsistent with every other search box on the site.
Why it matters
Users see a plain unstyled text box instead of the standard branded search input. The missing clear button is a usability loss on mobile.
Suggested fix
Change type="text" → type="search" on the #tc-search input. The shared CSS then applies automatically: magnifier icon, gold ::-webkit-search-cancel-button, and consistent focus styling. No JS changes needed — the existing filter logic works on both input types.
Inline <style> block duplicates shared pill and search CSS
City-Transit-Cards.html · <style> block · lines 9–130
What was found
The page defines .search-wrap, #tc-search, #tc-search::placeholder, #tc-search:focus, .pills, .pill, .pill:hover, and .pill.active inside a page-scoped <style> block. These duplicate classes that belong in shared CSS. When shared pill or search rules are updated in web-travel-style.css or mobile.css, this page will silently drift out of sync — the inline rules will win due to specificity.
Why it matters
Violates the site-wide SHARED CSS ONLY rule. Any future design system change (pill radius, active-state color, search width) will update every other page but leave the transit cards page visually stranded.
Suggested fix
Audit the inline block against web-travel-style.css and mobile.css. Move any rule that already exists in shared CSS (or belongs there) out of the inline block. Anything page-specific (e.g., transit-card-specific layout rules) may stay. Run python3 Brain/scripts/check_css_duplication.py to confirm exit 0 after the move.
Hardcoded hex colors in inline <style> block
City-Transit-Cards.html · <style> block
What was found
Two hardcoded hex values appear in the page's inline style rules where CSS variables should be used:
  • #tc-search::placeholder — color: #A8895A (should be var(--muted) or a placeholder token)
  • #tc-search:focus — border-color: #c8b99a (should be var(--border) or var(--accent) at reduced opacity)
Why it matters
Hardcoded colors don't respond to dark mode and can't be updated globally. Both values are close to existing tokens but not the same, creating subtle color inconsistency compared to other inputs on the site.
Suggested fix
Replace #A8895A with var(--muted); replace #c8b99a with var(--border2) or var(--accent). Verify in both light and dark modes.
Autocomplete attached to a type="text" input — clear button UX gap
City-Transit-Cards.html · inline <script> · TVESearch.attach(input, …)
What was found
The search-autocomplete.js script is loaded and TVESearch.attach() is called correctly on #tc-search. However, because the input is type="text", the native browser clear button (::-webkit-search-cancel-button) that the shared CSS styles gold is not present. Users who type a search term have no in-field clear affordance — they must backspace or click a filter pill to reset.
Why it matters
Minor usability gap, especially on mobile. Resolved as a side effect of fixing finding #1 (changing to type="search").
Suggested fix
No separate fix needed — this resolves automatically once the input type is changed to type="search" per finding #1 above.

Section 3 — Recommendations to improve the page

These are suggestions for the owner — not items the UX routine fixes automatically. Grouped by priority.

High priority
No intro / "how to use this page" block
Look & Content
The page opens directly with the search bar and region filters. There is no sentence explaining what a transit card is, why a visitor would want one instead of tapping a bank card, or how to use the filters. First-time visitors who land here from a guide or search have no context — they may not understand the difference between "Deposit" and "Card fee," or why some cities show 📲 and others don't.
Suggested improvement
Add a compact intro block (2–3 sentences) below the title, above the search bar: what transit cards are, when to get one vs. just tapping your bank card, and a one-line key for the 📲 badge. This doubles as useful SEO-adjacent context and helps mobile users who scroll past the filters.
Airport access is rarely mentioned in tip bars
Content gap — fleet-wide
Scanning all entries: only a handful of cities (Tokyo, Stockholm, Helsinki, Vancouver) mention whether the card covers the airport rail or requires a supplement. This is the #1 practical question travelers have — they arrive, want to take the train into the city, and don't know if their transit card works or if they need to buy a separate ticket. Cities like Paris (CDG — not covered by standard Navigo zones), Rome (Leonardo Express — not on the transit card), Barcelona (T-Casual — does NOT cover the airport), and Chicago (CTA Blue Line — does, no supplement) are currently silent on this.
Suggested improvement
Add a one-line airport note to the tip bar for every city that has a major international airport with a rail link. Use a consistent format: "✈️ Airport: [covered / zone add-on €X / separate ticket required]." This is high-value, immediately actionable information that no other element on the page currently provides.
App-only cities don't warn users to set up the app before arrival
Content — Copenhagen, Munich, Oslo
Copenhagen (Rejsekort App / Basiskort), Munich (MVV/MVGO app), and Oslo (Ruter app) all require downloading and setting up a mobile app to buy tickets — there is no tap-at-gate contactless option and no physical card sold at most points of entry. The current tip bars describe the apps but don't flag the critical implication: if a traveler's phone is dead, roaming is off, or the app store is slow at the airport, they have no way to board. This is a real gotcha that catches visitors off guard.
Suggested improvement
Add a bold "⚠️ Download before you arrive — no tap-to-pay at gates" line as the opening sentence of the tip bar for all app-only cities. Consider also adding an "app-only" filter tag to data-tags so users can filter for this condition.
Medium priority
Two-system cities don't clearly warn that one card ≠ full coverage
Content — Bangkok, Rio de Janeiro, Santiago, Buenos Aires (buses)
Bangkok requires a Rabbit Card for BTS and a bank card for MRT — two separate systems with no integration. Rio requires RioCard Mais and a separate Jaé card. Santiago's Pago Ágil works on metro but not buses. Buenos Aires contactless works on Subte + city buses but not suburban trains. The current entries describe both systems, but a traveler skimming the row might conclude "I can tap my bank card everywhere" when that is false. The visual layout presents both items side-by-side without a clear warning that neither covers the full network.
Suggested improvement
Open the tip bar for each two-system city with an explicit "⚠️ Two separate systems — no single card covers all lines." Then explain which card/method covers which system. Consider a visual separator or a distinct row style for multi-system entries so they stand out from single-card cities at a glance.
Thin tip bars on several recently added entries
Content — Dublin, Porto, Geneva, Istanbul, Montreal
Several of the newly added entries have tip bars with one sentence or less of actionable advice — enough to pass validation but not enough to actually help a traveler plan. Examples: Dublin's tip doesn't mention that the Leap Card gives a discount vs. cash fare; Porto's doesn't warn that the Andante card zones are confusing for visitors; Geneva's doesn't mention that the hotel transit card only covers the Canton of Geneva and not intercity trains; Istanbul's doesn't note that card is essential because cash fares cost significantly more; Montreal doesn't flag that the OPUS card can't be loaded via app and must be reloaded at a physical machine or station.
Suggested improvement
On the next update pass, expand tip bars for these five cities with at least 2–3 practical sentences each. The research is already done — these are well-documented transit systems. Prioritize the information travelers are most likely to need on arrival day.
Stat counter accuracy after large additions
Look — page header counter bar
The page shows a counter like "N cities · M regions · K tap-to-pay" calculated from data-tags in JavaScript. This run added 31 cities, bringing the total significantly higher. The counter itself is dynamic (computed at runtime) so it should be correct — but the number of regions is hardcoded as the count of .section divs, and a "regions" label that shows "4" may now feel low given the page covers ~55+ cities across 4 regions. The metric is correct but may not be the most impressive framing.
Suggested improvement
Consider changing the counter bar to show "N cities · K with tap-to-pay · 4 regions" or adding a "countries covered" count (derivable from the entries) instead of the region count, which will always be 4. The city count after this run is a strong headline number worth surfacing prominently.
Low priority
No "card expiry / value loss" information for any entry
Content gap — selected cities
Several transit cards expire or lose value if unused for a period (London Oyster balance is retained indefinitely but the card itself expires; Tokyo IC cards lose value after 10 years; Hong Kong Octopus has no expiry). Travelers who visit a city repeatedly often keep their transit card between trips — a one-line note on whether this is safe would be useful for returning visitors.
Suggested improvement
Add a brief "♻️ Reusable" or "⏳ Expires after X years" note to the tip bar for cities where this is commonly relevant (London, Tokyo, Hong Kong, Singapore, Sydney). Low effort, high value for repeat visitors.
No filter for "physical card required" vs "contactless OK" vs "app only"
Look — filter controls
The current filter pills are All / Europe / Asia / Americas / Oceania / 📲 Tap to pay. There is no way to filter for "I only want to tap my bank card" (contactless) vs "I need to get a physical card before arrival" vs "I need an app." As the page grows, these distinctions become more practically important — a business traveler who never carries extra cards wants a very different answer than a backpacker who pre-loads a physical card at each destination.
Suggested improvement
Consider adding two new data-tags values: app-only (no physical card, no tap-to-pay) and exposing them as filter pills. The tag infrastructure already supports this pattern — it's an additive change. Low development effort; high utility for power users of the page.
