What's Working Well
Country chip filter is fast and reliable
The chip-based country filter on each regional page highlights a single country across all sections instantly. No page reload, no URL change — great UX for comparing a specific country's metrics at a glance.
Travel-Stats FMAP integration is unique
Flight time from SEA, nonstop availability, and been/want status per destination is site-specific data that no other travel stats resource offers. The 745h 47m total flight time and "farthest been" metrics are genuinely delightful details.
Sunshine tables (monthly, city-level) beat annual averages
Monthly sunshine hour tables with real city-level data are more useful than a single annual average. The sun-table-wrap pattern with Jan–Dec columns plus an annual total is a strong design that other stats sites rarely match.
Build/validate pipeline keeps data integrity tight
build_travel_stats.py + validate_travel_stats.py + validate_us_stats.py catch FMAP drift before it can diverge from the guide index. The monthly routine catches section-level gaps like the French Polynesia dataset exclusions found this run.
Recommendations — Data Gaps
1
Europe-Stats is missing HDI — Asia-Stats and South-America-Stats both have it

The Human Development Index is one of the most cited cross-country metrics, and Asia-Stats and South-America-Stats both include it. Europe-Stats does not have an HDI section despite having 29+ countries. The UNDP Human Development Report 2024 covers all EU countries. The data exists; the section just hasn't been added.

Fix: add a hdi bar section to Europe-Stats between poverty and university, sourced from UNDP HDR 2024. Rank order: Ireland (1st in Europe), Switzerland (2nd), Norway (3rd), etc.

Effort: medium (add 30 bar rows + section header). Data source: UNDP HDR 2024.
2
Caribbean-Stats has no UNESCO Sites section

Caribbean-Stats covers 12 data categories but omits UNESCO World Heritage Sites, which other regional pages include. Several Caribbean/Central American destinations have significant UNESCO presence: Mexico (35 sites, 6th globally), Costa Rica (4 sites), Barbados (Garrison Historic Area, 2011), Virgin Islands (none formal), Cayman/Aruba/Sint Maarten/Curacao (zero sites — itself useful data to surface).

Fix: add a UNESCO section to Caribbean-Stats using WHOSIS/UNESCO data.

Effort: small (11 countries, data easily available from UNESCO website).
3
Stats-Across-US has no "Cost of Living" or "Housing" breakdown by region

The US stats page covers 21 categories but doesn't have a "median home price" or "rent" section — data that travel planning actually needs (especially for remote workers or long-stay travelers). The current housing category tracks homeownership rate, not housing cost. C2ER COLI and Zillow Research publish city-level median rent and median home price data.

Fix: add a median rent section (or median home price if more appropriate) using C2ER 2024 data. ~30 cities, same bar format.

Effort: medium. Data available from C2ER Cost of Living Index 2024 or Zillow Research.
Recommendations — UX & Navigation
4
No link from country bar rows to the city guide

A user reading Europe-Stats and spotting that Czech Republic has 17 UNESCO sites has no direct path to the Prague guide from that section. Every bar row in every regional page is a dead end — it shows the data but doesn't link anywhere. This is the highest-impact UX gap on the stats pages.

Fix: make the 🇨🇿 Czech Republic span a link (or add a small guide-link icon after it) pointing to the corresponding city guide. This requires a mapping from country chip key → guide URL, which already exists in the FMAP data. The implementation is straightforward: a small JS lookup that wraps bar-country text in an <a> tag when a matching guide exists.

Note: countries with multiple guides (multiple Spanish cities, multiple Italian cities) would need a disambiguation — link to the most prominent city or to a search results page filtered to that country.

Effort: medium. JS side is small; mapping table needs thought for multi-city countries.
5
Stats-Across-US could benefit from a regional grouping toggle

With 30 US cities, scrolling through the population or cost-of-living bar to find a specific region's cluster (Northeast vs South vs West) is slow. A chip filter grouping — "Northeast · Southeast · Midwest · Southwest · West Coast · Mountain · Hawaii" — would let a user instantly compare Charlotte vs Miami vs New Orleans without manually scanning.

Fix: add a region-filter row above the stats nav, similar to the country chip pattern on other pages. Tags could be data attributes on the bar rows, and the JS filter is already proven to work.

Effort: medium (tagging 30 cities + JS filter). Design is already established from the country chip pattern.
6
Regional pages are navigation islands — Travel-Stats doesn't link to them

Travel-Stats.html is the hub, but it has no "Explore by region" section that links to the 6 regional stats pages. A user viewing their Travel-Stats summary has to navigate manually to find Europe-Stats or Asia-Stats. The regional pages don't cross-link to each other either.

Fix: add a small "Explore Regional Stats" row to Travel-Stats.html with pill links to all 6 regional pages. Also add a "Back to Travel Stats" link at the top of each regional page. This is a 5-line HTML addition to each page.

Effort: small (pure link additions, no data or JS changes). High impact-to-effort ratio.
Recommendations — New Pages
7
No "Africa" stats page — Lagos (Nigeria) is the only African guide with no stats home

Lagos (Nigeria) has a shipped guide and an FMAP entry but no regional stats page covering Africa. Egypt is currently in Europe-Stats's "Africa" sub-section, Morocco is also in Europe-Stats's Africa section, and Seychelles is in Asia-Stats. This patchwork isn't coherent.

As the Africa guide count grows — Cape Town is in-build, potential future guides for Nairobi, Marrakech, Cairo etc. — a dedicated Africa-Stats page becomes increasingly appropriate. The page structure (same chip + bar pattern) is proven.

For now: at minimum, add Lagos to the existing Europe-Stats "Africa" sub-section (under Seychelles/Morocco/Egypt) with country-level Nigeria data. Longer term: new Africa-Stats.html page.

Effort: small (Lagos add to Europe-Stats Africa section now); medium (new Africa-Stats page once 4+ Africa guides ship).
8
"Compare across regions" view doesn't exist

A user comparing Tokyo vs Barcelona vs Buenos Aires for a potential move or long stay has to open three regional stats pages and manually note figures. There's no cross-regional comparison. The existing Guides-Index has a side-by-side city compare (up to 5 cities) for guide metadata, but not for the statistical metrics the regional pages surface.

A cross-regional stats comparison page or modal would be distinctive — build it off the already-collected FMAP + regional stats data, let the user pick 2–4 destinations from anywhere in the world, and render their key stats (cost of living, life expectancy, vacation days, sunshine, corruption) side by side.

Effort: high (significant JS + data engineering). High long-term value as a unique feature.
Recommendations — Data Freshness & Automation
9
Data sources are 2023–2024 with no per-section freshness indicator

The pages show a single "Updated July 2026" stamp, but the underlying data sources range from 2023 to 2024 (OECD, UNDP, WHR). A user seeing "Happiness Index" doesn't know if the score is 2023 or 2024 data — it's in the section source line, but that's easy to miss.

Fix: add a year badge to each section title in the sticky nav or section header — e.g., "Happiness · 2024" or a small "(2024)" suffix in the cat-sec-src span, made visually distinct from the source name. This makes the recency transparent without cluttering the layout.

Effort: small (CSS + text change per section header). No data changes required.
10
Sections that can be auto-generated aren't — adding data is fully manual

The bar widths and rankings in the regional stats pages are hand-calculated (each width:XX% is a manually derived percentage of the max value). When a new country is added, every section needs manual rank recalculation and bar width adjustment. This creates a high maintenance burden and drift risk.

Fix: build a build_regional_stats.py script (modeled on build_travel_stats.py) that reads a JSON data file and regenerates the bar sections. The JSON would contain the raw data values; the script would calculate ranks and percentages. One manual data-entry step to populate the JSON, then the script handles all the bar math.

This would immediately eliminate the current risk of off-by-one ranks after a country is added (observed in several sections where rank gaps or mismatches exist).

Effort: high (script build + JSON data migration). Pays off within 2–3 countries added.
