2. validate_continuity_render.py has no scheduled periodic fleet scan Open · P2 · New

Type: Scheduling gap Script: Brain/scripts/validate_continuity_render.py Risk: CSS regression to guide-style.css goes undetected between ship events

Gap validate_continuity_render.py renders every guide and measures actual pixel gaps in Extras-section continuous-card joins and stop-block booking-box stacks — the exact class of bug that static checks cannot catch, and that shipped fleet-wide twice in one session on 2026-07-12 (once as a CSS specificity miss, once as a margin asymmetry miss). Both times the static anchor check (validate_itinerary.py) passed cleanly.

Current invocations:

  • Interactive session start — guide_tools.py start Step 3b-2 runs validate_continuity_render.py --warn (best-effort, advisory). But scheduled/routine sessions skip the ritual entirely, so this doesn't run in crib sessions.
  • Guide ship gate — guide_tools.py line 3337 runs the script fleet-scoped in strict mode. Catches regressions when any guide is shipped. But if no guide ships on a given day (or a CSS push lands between ships), the check doesn't run.

The window of exposure: a commit that edits guide-style.css can be pushed (it passes pre_push_guard.py's brain_check gate), live on the site, and producing visible card gaps across all 220 guides — until the next guide ships. The 2026-07-12 incidents prove this is not a hypothetical failure mode.

Add python3 Brain/scripts/validate_continuity_render.py --warn as a step in the cross-page-consistency-audit (Saturday) or ux-format-audit (weekly). The script needs Playwright (same requirement as validate_mobile_render.py, which already runs in cross-page-consistency-audit). A weekly render sweep of all guides would catch CSS continuity regressions within days of them landing, not "at next ship".
