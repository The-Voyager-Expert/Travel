Climate-Finder.html — "Browse by Climate"

No city-name search entry point Low · by design
The page is a reverse-climate finder driven by a month picker + a daytime-high temperature band (two range sliders). There is no text search box — correct for the interaction model, so the search-standard sub-checks (placeholder ellipsis, magnifier override, gold clear ✕) are N/A. UX consideration only: a lightweight "jump to a city" typeahead could complement the slider filter for users who already know the city they want. Not a defect — leave to UX judgment.
Hardcoded accent hex in the page <style> block Medium
Brand accents are written as literals rather than CSS tokens: the terracotta gradient, #b85c2a, #a61c00 (hi), #3d5282 (lo), #3d3a32, #d8d4cc. Structural colors correctly use var(--surface,#fff) / var(--border2,#e6e2da) with fallbacks, but the accents have no token. Per the SHARED-CSS-ONLY rule these accents should resolve to shared tokens. Widespread pre-existing pattern; grouped here for the UX/CSS-consolidation pass.
.sibling-pill component defined locally in two pages Medium
The .sibling-pill nav-pill class is defined identically in both Climate-Finder.html and Weather.html page <style> blocks. A component used by 2+ pages belongs in shared CSS (web-travel-style.css) — check_css_duplication.py flags this. Consolidate to one shared definition.
✓ Compliant: viewport tag present · shared CSS at correct depth (../assets/web-travel-style.css + ../assets/mobile.css) · toolbar mount has data-depth="1" data-maxwidth="1040" · <h1> text-only, uppercase via banner standard, terracotta underline · page gutter 0 32px desktop / 0 12px mobile · pill/filter states all correct (white rest, terracotta gradient on .on/hover, #b85c2a segmented buttons, terracotta slider accent-color — no gold-fill, no beige rest).

Weather.html — "Weather by City"

City picker is a native <select> of 220 cities — no typeahead / search Review
City selection is a plain HTML <select class="wx-city"> dropdown listing all 220 cities alphabetically. At that count a native dropdown is slow to scan and has no incremental search on most browsers. The routine's "search input present" expectation is not met here. A searchable/typeahead city picker (or reusing the shared TVESearch autocomplete component) would materially improve findability. Structural UX decision — for the UX routine to weigh.
Page <style> uses hardcoded hex with no token fallbacks Medium
The entire widget stylesheet is literal hex — #c4b896, #b85c2a, #6b6860, #a61c00, #3d5282, #d8d5ce, etc. — none routed through CSS variables (unlike Climate-Finder, which at least keeps var(--x, fallback) for structural colors). JS also emits inline style= colors into the rendered panel. Stronger SHARED-CSS divergence than the sibling page; consolidate to tokens in the CSS pass.
✓ Compliant: viewport tag present · shared CSS at correct depth · toolbar mount data-depth="1" data-maxwidth="460" · <h1> text-only banner · pill states correct (.wx-mon.on/hover terracotta gradient, .wx-unit button.on #b85c2a, .sibling-pill white rest / terracotta hover) — no gold-fill, no beige rest · updated-stamp present and driven by _generated.
