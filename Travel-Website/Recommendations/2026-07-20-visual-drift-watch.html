⚠️ SPEC/CODE MISMATCH check_ham_menu_height spec says position:absolute;top:100% — toolbar.js uses position:fixed;bottom:0, which bypasses the height check

Rule 567 + brain_check docstring says:

Locked spec (2026-07-20): .tb-ham-menu: position:absolute; top:100%; NO height or max-height. The panel must be natural height (only as tall as its items) so it scrolls with the page and leaves no blank space below the last entry.

toolbar.js actually has:

'.tb-ham-menu{display:none;position:fixed;top:64px;left:0;right:0;bottom:0;' + 'background:#ffffff;...overflow-y:auto;-webkit-overflow-scrolling:touch;...}'

The code includes an explicit developer comment explaining the deliberate choice:

/* The menu is position:fixed so it stays fully on-screen as the user scrolls — items never disappear off the top. The toolbar (.tb) is NOT fixed (scrolls away as usual); only the open menu panel is fixed. overflow-y:auto scrolls inside the panel; body overflow:hidden locks page scroll so only the menu scrolls while it is open. */

What brain_check actually checks: check_ham_menu_height scans the .tb-ham-menu{…} CSS block for literal strings height: or max-height:. The current block contains neither — it uses bottom:0 within position:fixed instead. The check passes even though position:fixed; top:64px; bottom:0 is a de facto height-constrained overlay (effective height = viewport − 64px) with a white void below the last menu item if items don't fill the screen. This is exactly the class of bug Rule 567 was created to prevent.

  • Spec says: natural height, scrolls with the page
  • Implementation is: fixed full-screen overlay with internal scroll
  • Developer comment confirms the fixed approach is intentional
  • brain_check passes because no height:/max-height: keyword — bottom:0 is an unchecked equivalent constraint
Recommended resolution — two paths:
  1. If position:fixed is intentional (the developer comment says it is): Update Rule 567 spec and brain_check docstring to reflect "fixed full-screen panel with overflow-y:auto." Add a check that prohibits bottom:0 without overflow-y:auto (catches the void bug without rejecting the current deliberate design). Update the MANIFEST entry for ham-menu-natural-height.
  2. If position:absolute was truly required: fix toolbar.js to position:absolute;top:100% (natural height, scrolls with page), and add if "position:fixed" in block: fail(…) to the check.

Gap type: Brain-check gap — check_ham_menu_height validates keyword absence only. bottom:0 in a fixed-position context is an unchecked equivalent height constraint.
