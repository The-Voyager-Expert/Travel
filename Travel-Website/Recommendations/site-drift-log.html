How to use this file

This file is for recurring patterns — not one-time findings. If you ran into something once, log it to To Do List/To_Do_List.md instead.

Log here when a routine sees the same problem across multiple runs. That's drift: something that keeps coming back despite not being fixed. The pattern itself is the signal.

To add an entry: append a .log-row inside the matching .entry-log, or create a new .entry block if this pattern hasn't been logged before. Use the status badges: Watching (seen but not escalated), Escalated (logged to To Do List for action), Resolved (root cause fixed).

Toolbar version lag on auto-generated and static pages Escalated
Auto-generated pages (Travel-Stats.html, Safety-Guide.html) embed a stale toolbar.js?v=NNN version string baked into their build scripts. Every time toolbar.js bumps version, these pages fall behind and brain_check warns on them. Static pages similarly accumulate version lag between refreshes. The root cause is that the toolbar version string is duplicated in multiple build-script templates (build_travel_stats.py, build_safety_guide.py) rather than read from a single source. Seen across scripts-docs-audit, rule-compliance-audit, claudemd-drift-audit, validator-audit, and brainmd-audit.

verify-booking blind retry loop Escalated
When a guide's verification_log.json is empty or missing, guide_tools.py ship fails at verify-booking with no detail about which URLs are missing entries. Cribs have no visibility into the specific problem — they retry ship, get the same failure, retry again. Pattern: 3–8 FAIL @ verify-booking per guide before the crib figures out which bot-blocked URLs need log entries. Root cause: 2026-07-14 auto-prune incident stripped verification_log.json from 208 of 219 guides. Partial fix: check-vlog <City> command added to guide_tools.py (2026-07-21) so cribs can see missing entries before shipping. Fleet-wide log rebuild still pending.

High validate FAIL count per guide (structural errors) Watching
New guides consistently require 15–25 validate FAIL cycles before reaching PASS. Evidence: Bilbao 21 FAILs, Hilton Head 17 FAILs, Kotor 14+ FAILs. These are structural errors — wrong CSS class, missing wrapper div, wrong section order — not content errors. The fix <City> shortcut (~20s per cycle) reduces wall-clock cost but doesn't address the root cause. A guide stub template generator would eliminate structural FAILs at the source.

Trip-Essentials pages ship terracotta/temp hex literals instead of dark-mode tokens Escalated
Trip-Essentials interactive pages (Climate-Finder, Weather, Delta-Routes, Pickleball) keep shipping literal #b85c2a (terracotta accent), #a61c00/#3d5282 (hi/lo temp), and #4a7c9b (rain) hex values in their inline <style> blocks instead of the shared tokens --rust/--c-temp-hi/--c-temp-lo/--c-rain. These do not adapt in dark mode. Fixed piecemeal each UX run (Weather partly tokenized 2026-07-20, but its terracotta survived and re-surfaced 2026-07-22) then reappear. Root cause: brain_check.check_no_hardcoded_hex_colors's PALETTE dict lists only 14 NEUTRAL colors (#ffffff, #e6e2da, …) — it does NOT include the accent/temp/rain hexes that now have tokens, and its regex requires 6-digit hex so 3-digit #fff rest-states also slip through. The gate literally cannot catch these, so they recur.

Routine Artifact URLs die between runs (scheduled/headless sessions lose write access) Escalated
Shared Artifact URLs stored in memory for recurring routines keep returning "deleted, or you no longer have write access" when a later scheduled/headless session tries to update them in place. Three distinct URLs have died in ~24h: validator-coverage (`65a2e7cc…`), rolling-audit (`90e32eb3…`), and the shared weekly-ideas artifact (`1a641946…`). Each routine works around it by re-minting a fresh URL and rewriting its memory key, but that fragments the "one canonical URL" convention and means owner-facing bookmarks silently break. Underlying data is never lost (all reports also write to disk / GitHub Pages). Suspected root cause: artifacts published from automated sessions aren't retaining owner write-access for the next session.
