Fix 1: Global Non-ASCII Whitespace Check (Line 959)

What the Validator Index says

"Non-ASCII whitespace — global hard fail (locked 2026-05-18)" — mandates checking for U+2004 THREE-PER-EM, U+200A HAIR SPACE, U+00A0 NBSP, U+2009 THIN SPACE, U+202F NARROW NBSP, U+200B ZERO-WIDTH SPACE, and 7+ others.

What was actually implemented (the gap)

Only a narrow check existed: after 🚕 emoji in motion banner divs (.next, .hotel-first, etc.) for exactly U+2004 and U+200A. This left the following undetected:

  • Non-ASCII spaces appearing ELSEWHERE in the same motion banner row (e.g., between "3 min" and "→")
  • Non-ASCII spaces in any other guide section (stop boxes, extras, overview, etc.)
  • All other banned code points (U+00A0, U+2009, U+202F, U+200B, U+205F, U+3000, U+FEFF, U+2000–U+2003, U+2005–U+2008)

Fix applied

New check scans the full visible guide text (all HTML tags and comments stripped) for any of 17 banned Unicode space characters. One context example is reported per unique code point found. Carve-out preserved: U+200A immediately after ❗/❗️ in Heads Up titles is exempt (already documented at line 23156 of the validator).

Immediate real-world catch: Singapore v3 (singapore_v1.html) fails the new check with a genuine U+200A (HAIR SPACE) in a 🚕 3 min → Marina Bay Sands SkyPa... motion banner row — the space is BETWEEN words, not immediately after the 🚕 emoji, so the old narrow check missed it. This confirms the gap was real, not theoretical.

Recommended follow-up

Singapore v3 needs a content fix: locate the U+200A HAIR SPACE in the motion banner row and replace with a standard U+0020 ASCII space. Run validate_itinerary.py singapore_v1.html to confirm the exact location. This fix is out of scope for the audit crib (own-work rule) — assign to the next Singapore-touching session.
✓ Fixed 2026-07-18 by pending-fixes-sweep — 25 U+200A hair spaces replaced with ASCII space across all motion banners; guide re-validated at 831 passed / 0 failed and re-stamped.

Fix 2: Global Inline style= on <div> Check (Line 960)

What the Validator Index says

"Global inline style= on div — previously only font-size, font-family, and text-decoration were individually checked — this global check catches everything else" — mandates checking for: margin, padding, border, background, color, display, width, height, opacity, flex, grid, position, z-index.

What was actually implemented (the gap)

Per-class inline style checks existed (e.g., checking style="display:none" in specific contexts, checking style="color:..." on .dep/.arr spans). No global scan of ALL <div> elements for layout/paint inline styles existed.

Fix applied

New check scans every <div style="..."> in the guide HTML for inline styles containing any of the 13 banned layout/paint properties. One sentinel exemption: display:none alone is allowed (used as a JavaScript-controlled visibility toggle in guide HTML). Preventive guard — 0 current guides are affected.
