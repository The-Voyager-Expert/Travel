🔏 Validation stamp — signed proof (validate locally, verify everywhere)

The validation stamp is content-bound and HMAC-SIGNED (Brain/scripts/validation_stamp.py): <!-- validation: passed YYYY-MM-DD HH:MM | v2 | sig=<hmac> --> where sig = HMAC-SHA256(key, the guide's whitespace-collapsed content minus the stamp). The 400-rule validator runs locally and writes this stamp only on a clean pass; every other gate just verifies the signature — a cheap hash, never a re-run of validation. A hand-typed / bulk-written stamp has no matching signature, and a guide edited after validation no longer matches its signature — both are rejected. (reimplemented 2026-06-27, replacing the forgeable plain-string stamp.)

  • validate_itinerary.py (local) — on a clean pass writes the signed stamp via validation_stamp.write_stamp(). Runs anytime on one guide.
  • guide_tools.py ship (local) — re-runs the validator on this one guide (re-signs), then validation_stamp.verify() hard-gates before the queued push. Validates only its own guide — never the fleet.
  • pre_push_guard.py (local hook) — blocks a push that adds/modifies a guide lacking a valid signed stamp. --no-verify bypassable. Safe-mechanical-update bypass (added 2026-07-14): when an unstamped guide's only diff vs origin is confined to a registered exempt block, the guard auto-re-stamps it instead of blocking — no full re-validation needed. The exempt blocks live in _SAFE_BLOCKS (open/close marker comments); _is_safe_mechanical_diff() confirms every changed line sits inside one, then _auto_restamp() re-signs. First registered block: <!-- nearby-guides --> … <!-- /nearby-guides --> (the build_nearby_guides.py injection, already exempt from every TR-* check). Any change outside a safe block still forces normal validation. To add a new mechanical injector: register its marker pair in _SAFE_BLOCKS.
  • Deploy = publish only. The deploy (.github/workflows/deploy-pages.yml) does NOT validate, re-validate, verify, or quarantine — it stages Travel-Website/ + .nojekyll and publishes. The repo only ever contains validated guides (a failing guide stays untracked on the Drive; the pre-push hook blocks it), so there is nothing to check at deploy. No CI validation, no placeholders, no all-or-nothing block. (Dani 2026-06-27 — the quarantine deploy gate was removed.)
