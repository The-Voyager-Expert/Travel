Summary: A shared navigation bar rendered at the top of every page. Links all Trip Essentials files and the Guides index so the reader can navigate without returning to a root page. The active page highlights automatically. The toolbar reads the page background color at runtime and adapts its accent — no per-page style override is needed.

§ 1. Required markup

Place the mount div and script tag immediately after the opening <body> tag, before any other content. Both are required on every new page.

<div id="toolbar-mount" data-depth="N" data-maxwidth="W"></div> <script src="PATH-TO-assets/toolbar.js"></script>

Omitting the mount div silently skips the toolbar. Omitting either attribute produces a mis-sized or broken navigation bar.

Optional attribute: data-toolbar-theme="guide" forces the guide accent color on pages that do not have the guide background color — used on the Guides index page, which has a non-guide background but should visually match the guide family. Omit this attribute on all other pages; the toolbar detects the accent automatically from the page background.

§ 2. data-depth

Set to the number of directory levels the page sits below the site root (Travel-Website/). The toolbar uses this value to build relative hrefs to every other page and to load weather.js from assets/. (footnote.js is no longer auto-loaded — [RETIRED 2026-06-06]; toolbar.js carries a FOOTNOTE_RETIRED = true guard.) data-depth describes the page's location — not the script path. The script src points to assets/toolbar.js at the matching depth (the assets' permanent home):

  • 0 — the site home Travel-Website/index.html at the Travel-Website/ root. Script path: assets/toolbar.js
  • 1 — pages directly inside Trip-Essentials/, and Guides/Guides-Index.html (e.g. Trip-Essentials/Travel-Packing.html). Script path: ../assets/toolbar.js
  • 2 — guide pages inside Guides/{City}/, and pages one subfolder deeper (e.g. Trip-Essentials/Maps/World-Map.html, Trip-Essentials/Plug-Adapter/Plug-Adapter-Guide.html). Script path: ../../assets/toolbar.js

The src attribute on the <script> tag is the relative path from the page to assets/toolbar.js — it mirrors data-depth (depth ../ segments + assets/). Never load the toolbar from Guides/ or the Travel root; the assets do not live there.

§ 3. data-maxwidth (legacy — no longer caps the bar)

Historically this set a fixed column width for the button row. It no longer controls the bar's width. The toolbar self-centers on the viewport axis (the same axis the page content centers on) via width:max-content + margin:0 auto — see § 7 Centering. The attribute is still read for backward-compatibility and may stay on existing pages, but its value is inert; new pages do not need to tune it. Do not reinstate a width cap from this value — capping is exactly what broke centering (twice). Conventional values that remain on pages:

  • 760 — Trip Essentials pages
  • 940 — Guide pages

§ 4. Adding a page to the menu

The menu is defined in the ITEMS array inside Travel-Website/assets/toolbar.js. To add a new page, add one entry to that array. One edit propagates to every page that loads the script — do not add link markup to individual HTML files.

⚠️ This is the stable original toolbar — do NOT redesign the desktop look or behaviour without an explicit owner request. A 2026-06-20 redesign (consolidated groups, larger type, new layout) was reverted at the owner's direction: "the desktop toolbar should not have changed." Add or remove pages only; leave the chrome alone.

Three entry forms. A plain link { href, text } (href = path relative to the site root Travel-Website/, URL-encoded; text = label); a dropdown group { group: '💰 Money', children: [ { href, text }, … ] } — one toolbar button that opens a flyout of its child links (the parent has no href of its own); or null — a thin separator between groups of links. On desktop a flyout is appended to <body> and fixed-positioned on open (escapes the scroll row's overflow clip); only one flyout is open at a time; the group highlights active when the current page is one of its children; a click anywhere else closes it. The compact 13px row centers on the viewport and scrolls horizontally if it is wider. On mobile the toolbar collapses to a hamburger with a flat link list (groups expanded inline). Groups are one level deep — no nested sub-groups. Do not add link markup to individual HTML files — one edit to the array propagates to every page.

Live structure (owner-curated). The exact top-level arrangement is the owner's call and changes from time to time; the ITEMS array in toolbar.js is the source of truth. As of 2026-06-20 the bar is (separators between groups):

  • 📆 Trips ▾ — Trips · Packing

Every page the site owns is still reachable (no page dropped — only regrouped). The whole toolbar uses one pinned font stack and a single size per surface, so labels never change size between the bar, the flyouts, the mobile accordion, or between pages (a map page no longer renders the bar in a different font). brain_check (check_toolbar_page_standard / check_toolbar_pages_documented) reads every href: base + '…' in the ITEMS block, including group children, so each child page is gated to the shared standard and must be documented in Brain.md Part 1.

§ 5. Visual design (current values — 2026-06-13)

All visual values live in toolbar.js. Do not override per-page.

  • Bar background: rgba(245,244,240,.96) — a warm off-white band that matches the page; near-opaque, not transparent
  • Bar border: none
  • Bar shadow: none
  • Bar padding: 10px 0 desktop · 6px 0 4px mobile
  • Desktop links: no border, transparent background — they are plain text links, not bordered pills. font-size:11.5px, text #3d3a32, padding:4px 9px, border-radius:4px (rounds only the hover/active background, not a visible box).
  • Desktop hover: text switches to the accent color with a faint acLt background tint (auto-detected per page type).
  • Active link (desktop): accent color text, acMd background tint, font-weight:500.
  • Mobile links: plain text, no border, no background — font-size:12px, color:#5a5650, natural wrap, min-height:44px tap target. Active link uses accent color + font-weight 500 only.
  • Scroll behavior: horizontal-only via inner.scrollLeft — never calls scrollIntoView to avoid vertical page jump on load. The bar only scrolls when the buttons are wider than the whole window (very narrow screens); on normal laptops every button is visible — see § 7 Centering

§ 7. Centering — LOCKED (validator-enforced)

The button row must stay centered on the same axis as the page content, on every browser and OS, and must never hide the first button. The page content is centered in the viewport (.wrap / main are margin:0 auto), so the toolbar must center in the viewport too.

Locked mechanism (in toolbar.js, do not change without re-reading this section):

  • .tb-links (the button row) — width:max-content + margin:0 auto. The row is exactly as wide as its buttons and centers in the viewport. No width cap.
  • .tb-inner (the scroll container) — full width, overflow-x:auto. It scrolls only when the buttons exceed the whole window.
  • No max-width / fit-content on the row; no maxWidth applied to the scroll container in JS.

Why this is locked — two regressions, both about width-capping:

  • Cap on the row + justify-content:flex-start: button width depends on the browser/OS font and emoji set (macOS Apple Color Emoji are wider). When the buttons exceeded a fixed cap, the row froze at the cap and packed every button to the left, leaving a gap on the right. Looked fine on some machines, broken on others.
  • Cap on the scroll container (scroller.style.maxWidth): the bar scrolled, and the "scroll active tab into view" logic re-centered the active tab — pushing the first button (Trips) off the left edge.

A row sized to its content and centered in the viewport has neither failure: it can't left-pack (it's exactly as wide as the buttons) and it doesn't scroll on a normal screen (so the active-tab scroll never hides Trips).

Enforcement: brain_check.py → check_toolbar_centering() hard-fails if toolbar.js drifts — i.e. if the row loses width:max-content or margin:0 auto, or if a width cap (fit-content, max-width on the row, or scroller.style.maxWidth) is reintroduced. It runs at every session start and as part of guide_tools.py ship. data-maxwidth is legacy and inert (see § 3).

§ 8. Weather widget (weather.js + climate.json) — added 2026-06-13

Note — paths updated for the 2026-06-13 site reorg. The shareable site now lives under Travel/Travel-Website/ and the shared assets (toolbar.js, weather.js, climate.json, guide_v3.css, mobile.css, footnote.js) all live in Travel/Travel-Website/assets/. Pages load them at their own relative depth below the site root.

What it is: 🌡 Weather — a white inline-SVG thermometer (filled with currentColor so it reads on the banner's dark background) followed by the word, as plain clickable text, no button chrome. It lives only on the Guides index, right-aligned inside the "Travel Guides" title banner (fills the banner's empty space — no wasted line — and stays clear of the search box). Clicking it opens a panel with a city picker (any of the cities), the typical high / low for any month, a 12-month range strip, and a °C / °F toggle (remembered via localStorage).

Per-guide hover. On the index, hovering any destination card shows a small popover with that city's typical weather — current-month high/low plus a compact 12-month strip, in the chosen unit. The city is resolved from the card's href folder, matched against the baked-in data.

Not shown on guide pages. toolbar.js injects weather.js only when curr === 'Guides-Index.html'; weather.js also self-guards (if (!isIndex) return). The banner control appends into .header (sets it to flex space-between; falls back to a fixed top-right position if the header isn't found) and does NOT touch the search box or topbar. The index defaults to the "By country" view on load.

Climate normals, not a forecast. The numbers are historical monthly averages (avg daily high/low), so they do not shift — unlike the on-demand forecast (On Demand/Weather - On Demand.html, which still governs live forecasts and stays chat-only). The two never overlap: normals = the always-on widget; forecast = on-demand, surfaced in chat near a trip.

Files — permanent home inside assets/:

  • Travel/Travel-Website/assets/weather.js — the widget. The climate data is BAKED INTO this file (a CLIMATE object between /* CLIMATE_DATA_START */ … /* CLIMATE_DATA_END */ markers) — no fetch, no network. This is deliberate: pages are opened both from the live site and as local files straight from Drive (file://), where a fetch() of a sibling .json is blocked by the browser. Inlining the data makes it work everywhere.
  • Travel/Travel-Website/assets/climate.json — the canonical data record: one entry per guide folder, { city, lat, lon, hi[12], lo[12] }, twelve monthly averages Jan…Dec in °C, plus a _meta block. The widget does NOT read this at runtime; it is the source of truth that the generator also bakes into weather.js.

Loader: toolbar.js injects weather.js as {base}assets/weather.js?v=N only on Guides-Index.html — the same loader pattern the retired footnote feature used. weather.js carries no version tag on the page itself, so bump the ?v= in that loader line (and the index's toolbar.js?v=) whenever the toolbar/weather behavior changes, so the browser refreshes.

Regenerate the data: python3 Brain/scripts/build_climate.py. It reads the coordinates from the Travel-Website/Trip-Essentials/Maps/ PINS arrays (three pinless cities seeded in the script), pulls Open-Meteo historical normals, rewrites assets/climate.json, AND re-bakes the CLIMATE block inside assets/weather.js between its markers (and the same block in Trip-Essentials/Trips.html, which powers the Next-up countdown's destination weather). Run it after adding new guides so the new city gets weather. Open-Meteo rate-limits bursts (HTTP 429); the script paces and is resumable.

§ 9. PWA — install to home screen + offline

What it is (added 2026-06-15): the site is an installable Progressive Web App. On iPhone Safari, Share → Add to Home Screen drops a Travel icon that launches fullscreen (no Safari chrome) and reads offline. Two parts power it, both wired from toolbar.js so every page gets them with no per-page markup.

Manifest + Apple tags. The PWA IIFE at the top of toolbar.js injects, into each page's <head> at the page's own data-depth: <link rel="manifest" href="{base}manifest.webmanifest">, apple-touch-icon, a 32px icon, theme-color (#b85c2a), and the apple-mobile-web-app-* / mobile-web-app-capable tags. It never double-injects (guards on existing tags).

Service worker — offline. The same IIFE registers {base}sw.js (only on https: / localhost; no-ops on file://). sw.js lives at the site root (Travel-Website/sw.js) so its scope covers every page — a worker can only control its own directory and below. Strategy: navigations are network-first with an offline cache fallback; assets are stale-while-revalidate (instant from cache, refreshed in the background). It caches as you visit — any page opened once is then readable with no signal. Bump CACHE in sw.js to force-clear old caches after a big change.

Files (permanent homes):

  • Travel/Travel-Website/manifest.webmanifest — app name, display:standalone, colors, start_url = Guides/Guides-Index.html, icon set. Paths are relative to the manifest (site root).
  • Travel/Travel-Website/sw.js — the offline service worker. Must stay at the site root for scope; never move it into assets/.
  • Travel/Travel-Website/assets/icons/ — app icons: apple-touch-icon.png (180), icon-192.png, icon-512.png, icon-512-maskable.png, favicon-32.png, plus app-icon-pin.svg (source). Full-bleed cream #faf7f0 with the terracotta #b85c2a location-pin mark; iOS applies the rounded mask. Alternate concepts (compass, globe) are archived in Icons Library/App Icon Concepts/.

§ 10. Last-updated stamp — guide pages (added 2026-06-15)

What it is: "Updated Month Year" appears in small muted text (font-size:11px, color:#b0aaa0) centered at the very bottom of every individual city guide page. It is injected by toolbar.js automatically — no per-guide markup required.

How it works: toolbar.js reads document.lastModified, which reflects the HTTP Last-Modified response header (file mtime when served from GitHub Pages, file mtime on local file:// preview). A year ≤ 2000 guard suppresses the stamp on pages that return a broken or absent header.

Page detection: the stamp fires only on depth-2 individual city guide pages. Detection regex in toolbar.js: /\/Guides\/[^/]+\/[^/]+\.html/. Trip Essentials pages, guides_index, and all other depth-0/1 pages are excluded.

Two-type update behavior — content vs format:

  • Content change (editing a stop, restaurant, hotel, or any text inside the guide HTML itself) → the guide file's mtime updates automatically → document.lastModified reflects the new date → stamp is accurate with zero extra work.
  • Format change (editing guide_v3.css, toolbar.js, or any shared asset) → the guide HTML file is never touched → mtime stays → stamp does not update. This is intentional. "Updated Month Year" means when the guide's content was last refreshed, not when a CSS tweak or toolbar feature was added. Never touch a guide HTML solely to bump the stamp date.

No build step, no validator check: the stamp is fully automatic — it requires no build action, no per-guide attribute, and no validator enforcement. It simply reflects the guide file's own last-written timestamp.
