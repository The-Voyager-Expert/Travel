§ 1. Required markup

Place the mount div and script tag immediately after the opening <body> tag, before any other content. Both are required on every new page.

<div id="toolbar-mount" data-depth="N" data-maxwidth="W"></div> <script src="PATH-TO-assets/toolbar.js"></script>

Omitting the mount div silently skips the toolbar. Omitting either attribute produces a mis-sized or broken navigation bar.

All content pages — add data-updated to the mount div. Guides use data-updated="YYYY-MM-DD" (e.g. data-updated="2026-07-24") — toolbar.js renders the day. Non-guide pages (Trip-Essentials, Best-Of, maps) use data-updated="YYYY-MM" (e.g. data-updated="2026-07") — no day shown. The attribute powers the "Updated" stamp at the top of the page (§ 10). TB-11 hard-fails a guide missing it; non-guide pages warn if absent. Bump only when a routine reviews or changes the page's travel content — never for bug fixes, CSS/JS changes, new toolbar features, new pages added to the site, or deploys. Companion pages (stops-map, read-about) omit it.

Optional attribute: data-toolbar-theme="guide" forces the guide accent color on pages that do not have the guide background color — used on the Guides index page, which has a non-guide background but should visually match the guide family. Omit this attribute on all other pages; the toolbar detects the accent automatically from the page background.

§ 2. data-depth

Set to the number of directory levels the page sits below the site root (Travel-Website/). The toolbar uses this value to build relative hrefs to every other page and to load weather.js from assets/. (footnote.js was removed 2026-06-21 — see § 6.) data-depth describes the page's location — not the script path. The script src points to assets/toolbar.js at the matching depth (the assets' permanent home):

  • 1 — pages directly inside Trip-Essentials/, and Guides/Guides-Index.html (e.g. Trip-Essentials/Travel-Packing.html). Script path: ../assets/toolbar.js
  • 2 — guide pages inside Guides/{City}/, and pages one subfolder deeper (e.g. Trip-Essentials/Maps/World-Map.html, Trip-Essentials/Plug-Adapter/Plug-Adapter-Guide.html). Script path: ../../assets/toolbar.js

The src attribute on the <script> tag is the relative path from the page to assets/toolbar.js — it mirrors data-depth (depth ../ segments + assets/). Never load the toolbar from Guides/ or the Travel root; the assets do not live there.

§ 3. data-maxwidth (legacy — no longer caps the bar)

Historically this set a fixed column width for the button row. It no longer controls the bar's width. The toolbar self-centers on the viewport axis (the same axis the page content centers on) via width:max-content + margin:0 auto — see § 7 Centering. The attribute is still read for backward-compatibility and may stay on existing pages, but its value is inert; new pages do not need to tune it. Do not reinstate a width cap from this value — capping is exactly what broke centering (twice). Conventional values that remain on pages:

  • 760 — Trip Essentials pages
  • 940 — Guide pages

§ 4. Adding a page to the menu

The menu is defined in the ITEMS array inside Travel-Website/assets/toolbar.js. To add a new page, add one entry to that array. One edit propagates to every page that loads the script — do not add link markup to individual HTML files.

⚠️ This is the stable original toolbar — do NOT redesign the desktop look or behaviour without an explicit owner request. A 2026-06-20 redesign (consolidated groups, larger type, new layout) was reverted at the owner's direction: "the desktop toolbar should not have changed." Add or remove pages only; leave the chrome alone.

Three entry forms. A plain link { href, text } (href = path relative to the site root Travel-Website/, URL-encoded; text = label); a dropdown group { group: '💰 Money', children: [ { href, text }, … ] } — one toolbar button that opens a flyout of its child links (the parent has no href of its own); or null — a thin separator between groups of links. On desktop a flyout is appended to <body> and fixed-positioned on open (escapes the scroll row's overflow clip); only one flyout is open at a time; the group highlights active when the current page is one of its children; a click anywhere else closes it. The 14px row centers on the viewport; .tb-inner uses overflow-x:hidden so it does not scroll horizontally (changed 2026-07-21 — owner decision). On mobile the toolbar collapses to a hamburger with a flat link list (groups expanded inline). Groups are one level deep — no nested sub-groups. Do not add link markup to individual HTML files — one edit to the array propagates to every page.

§ 7. Centering — LOCKED (validator-enforced)

The button row must stay centered on the same axis as the page content, on every browser and OS, and must never hide the first button. The page content is centered in the viewport (.wrap / main are margin:0 auto), so the toolbar must center in the viewport too.

Locked mechanism (in toolbar.js, do not change without re-reading this section):

  • .tb-links (the button row) — width:max-content + margin:0 auto + justify-content:flex-start. The row is exactly as wide as its buttons and centers in the viewport by margin. justify-content:flex-start ensures overflow falls off the right — so Guides (first item) is never pushed into negative-scroll territory. Do not use justify-content:center — with no width:max-content, center overflows on both sides and hides Guides to the left of scroll position 0 (regressed 2026-07-19, fixed same day). No width cap.
  • .tb-inner (the scroll container) — full width, overflow-x:hidden (changed from auto 2026-07-21 — toolbar no longer slides horizontally). The width:max-content + margin:0 auto centering on .tb-links works the same whether the container scrolls or clips — the row is sized to its buttons and centered; on any normal screen the row fits without overflow.
  • No max-width / fit-content on the row; no maxWidth applied to the scroll container in JS.

Why this is locked — two regressions, both about width-capping:

  • Cap on the row + justify-content:flex-start: button width depends on the browser/OS font and emoji set (macOS Apple Color Emoji are wider). When the buttons exceeded a fixed cap, the row froze at the cap and packed every button to the left, leaving a gap on the right. Looked fine on some machines, broken on others.
  • Cap on the scroll container (scroller.style.maxWidth): the bar scrolled, and the "scroll active tab into view" logic re-centered the active tab — pushing the first button (Guides) off the left edge.

A row sized to its content and centered in the viewport has neither failure: it can't left-pack (it's exactly as wide as the buttons) and it doesn't scroll on a normal screen (so the active-tab scroll never hides Guides).

Enforcement: brain_check.py → check_toolbar_centering() hard-fails if toolbar.js drifts — i.e. if the row loses width:max-content or margin:0 auto, or if a width cap (fit-content, max-width on the row, or scroller.style.maxWidth) is reintroduced. It runs at every session start and as part of guide_tools.py ship. data-maxwidth is legacy and inert (see § 3).

What it is: 🌡 Weather — a white inline-SVG thermometer (filled with currentColor so it reads on the banner's dark background) followed by the word, as plain clickable text, no button chrome. Per-guide hover. On the index, hovering any destination card shows a small popover with that city's typical weather — current-month high/low plus a compact 12-month strip, in the chosen unit. The city is resolved from the card's href folder, matched against the baked-in data.

Not shown on guide pages. toolbar.js injects weather.js only when curr === 'Guides-Index.html'; weather.js also self-guards (if (!isIndex) return). The index defaults to the "By country" view on load.

Climate normals, not a forecast. The numbers are historical monthly averages (avg daily high/low), so they do not shift — unlike a live on-demand forecast check (chat-only, surfaced near a trip — see Trip-Essentials/Weather.html on the site). The two never overlap: normals = the always-on widget; forecast = on-demand, surfaced in chat near a trip.

Files — permanent home inside assets/:

  • Travel/Travel-Website/assets/weather.js — the widget. The climate data is BAKED INTO this file (a CLIMATE object between /* CLIMATE_DATA_START */ … /* CLIMATE_DATA_END */ markers) — no fetch, no network. This is deliberate: pages are opened both from the live site and as local files straight from Drive (file://), where a fetch() of a sibling .json is blocked by the browser. Inlining the data makes it work everywhere.
  • Travel/Travel-Website/assets/climate.json — the canonical data record: one entry per guide folder, { city, lat, lon, hi[12], lo[12] }, twelve monthly averages Jan…Dec in °C, plus a _meta block. The widget does NOT read this at runtime; it is the source of truth that the generator also bakes into weather.js.

Loader: toolbar.js injects weather.js as {base}assets/weather.js?v=N only on Guides-Index.html — the same loader pattern the retired footnote feature used. weather.js carries no version tag on the page itself, so bump the ?v= in that loader line (and the index's toolbar.js?v=) whenever the toolbar/weather behavior changes, so the browser refreshes.

Regenerate the data: python3 Brain/scripts/build_climate.py. It reads the coordinates from the Travel-Website/Trip-Essentials/Maps/ PINS arrays (three pinless cities seeded in the script), pulls Open-Meteo historical normals, rewrites assets/climate.json, AND re-bakes the CLIMATE block inside assets/weather.js between its markers (and the same block in Trip-Essentials/Trips.html, which powers the Next-up countdown's destination weather). Run it after adding new guides so the new city gets weather. Open-Meteo rate-limits bursts (HTTP 429); the script paces and is resumable.

§ 9. PWA — install to home screen + offline

What it is (added 2026-06-15; install banner added 2026-07-19): the site is an installable Progressive Web App. Users can save it to their mobile home screen — it launches fullscreen with no browser chrome and reads offline. Everything is wired from toolbar.js so every page gets the full PWA treatment with no per-page markup.

App name (updated 2026-07-19): manifest.webmanifest name = "The Voyager Expert"; short_name = "TheVoyagerExp" (what appears under the icon on the home screen). The apple-mobile-web-app-title meta tag (injected by toolbar.js) mirrors the short name. To rename: change both the manifest and the meta('apple-mobile-web-app-title', …) line in the PWA IIFE.

Manifest + Apple tags. The PWA IIFE at the top of toolbar.js injects, into each page's <head> at the page's own data-depth: <link rel="manifest" href="{base}manifest.webmanifest">, apple-touch-icon, a 32px icon, theme-color (#b85c2a), and the apple-mobile-web-app-* / mobile-web-app-capable tags. It never double-injects (guards on existing tags).

Service worker — offline. The same IIFE registers {base}sw.js (only on https: / localhost; no-ops on file://). sw.js lives at the site root (Travel-Website/sw.js) so its scope covers every page. Strategy: network-first for every request — online users always get the latest from the server; the cache is an offline fallback only (any page visited once stays readable with no signal). A MIN_VERSIONS map rewrites stale ?v= URLs so iOS HTTP cache is bypassed. Bump CACHE in sw.js AND the relevant MIN_VERSIONS entry to force-clear old caches and pull new assets after a significant change.

In-page install banner (added 2026-07-19). A second IIFE in toolbar.js shows a bottom-sheet banner on mobile — the in-page "tap to install" experience that mirrors how apps prompt. Two paths:

  • Android / Chrome: intercepts the browser's beforeinstallprompt event (fires when Chrome decides the PWA criteria are met — repeated visits, time-on-site, valid manifest). Banner slides up ~2 s after the event fires. The Add button (terracotta) calls deferredPrompt.prompt(), which opens the native install dialog. After the user chooses, the banner dismisses.
  • iOS / Safari: no beforeinstallprompt event exists. Instead, the IIFE detects iPhone/iPad Safari (UA check; excludes Chrome-on-iOS / Firefox-on-iOS) and shows the same banner with a manual instruction: "Tap 📤 Share then Add to Home Screen." Appears ~3 s after page load.

Banner behavior: shown only on mobile viewports (≤768 px), only on HTTPS, and only when the page is not already running as a standalone PWA (display-mode: standalone / navigator.standalone). Dismiss (× button) sets localStorage key tve_a2hs_dismissed → banner never appears again on that device. To reset for testing: open DevTools → Application → Local Storage → delete tve_a2hs_dismissed, or use a private/incognito tab.

Files (permanent homes):

  • Travel-Website/manifest.webmanifest — app name ("The Voyager Expert" / "TheVoyagerExp"), display:standalone, colors, start_url = Guides-Index.html, icon set. Paths are relative to the manifest (site root).
  • Travel-Website/sw.js — the offline service worker. Must stay at the site root for scope; never move it into assets/.
  • Travel-Website/assets/icons/ — app icons: apple-touch-icon.png (180×180, shown in the install banner + iOS home screen), icon-192.png, icon-512.png, icon-512-maskable.png, favicon-32.png, plus app-icon-pin.svg (source). Full-bleed cream #faf7f0 with the terracotta #b85c2a location-pin mark; iOS applies the rounded mask. Alternate concepts archived in Icons Library/App Icon Concepts/.

§ 10. Last-updated stamp — all content pages (added 2026-06-15; moved to top + made explicit 2026-06-21; extended to non-guide pages 2026-07-22)

What it is: "Updated Month Year" appears in small muted text (font-size:13px — matching the title address — color:#9a948a) right-aligned at the top of every city guide, tucked under the title block as the last row of .title-page (injected with flex order:6 + class="title-updated"; runtime-only). On Trip-Essentials and Best-Of pages it renders in the same position. Earlier (2026-06-15 → 2026-06-21) it sat centered at the very bottom in 11px #b0aaa0.

How it works: toolbar.js reads an explicit data-updated="YYYY-MM" attribute on the toolbar-mount div and renders it as "Updated {Month} {Year}". A malformed value or a year ≤ 2000 / month out of 1–12 suppresses the stamp; no attribute → no stamp (silent). It no longer reads document.lastModified.

On mobile (≤600 px) toolbar.js's repositionUpdatedStamp() moves it out of the top banner into the bottom-left of the guide's "Also on this site" card (left-aligned, 11px, muted — changed from right-aligned 2026-07-24 to avoid overlapping the "N days" pill), so the top banner stays clean and the hotel name/address card sits glued directly under the gradient line. Two sibling mobile-only movers run alongside it: repositionReadAbout() pushes the READ ABOUT link to the bottom of the Trip Overview section, and shortenTrainPill() relabels the "European Train Guide" resource pill to "European Train" for display only — the HTML keeps the canonical "European Train Guide" so check_also_on_site_pill_labels + validate_itinerary TR-4/TR-6 still pass (do not edit the guides' HTML for this). Full record: memory mobile_ui_format_2026_07.

Why explicit, not document.lastModified (changed 2026-06-21): lastModified reflects the file's mtime / HTTP Last-Modified header, which bumps on every touch — a bug fix, a validation re-stamp, even a GitHub Pages redeploy (Pages reports the deploy time, so all guides read the same deploy month). That is the opposite of content currency. The explicit attribute only moves when a crib deliberately bumps it.

Update behavior — the rule:

  • Content reviewed or refreshed (a routine audits a page's content, or you add/swap/rewrite a stop, restaurant, hotel, rate, or any substantive information) → bump data-updated to the current YYYY-MM in the same edit. This is a manual step and it is the only thing that moves the stamp. The bump must happen even on a clean pass — if the content was reviewed and found accurate, that review is still an event worth recording.
  • Bug fix / cosmetic / format change (typo, broken-link repair, icon tweak, CSS or toolbar.js edit, validation re-stamp) → do NOT bump data-updated. The stamp means "when the content was last reviewed by a crib," not "last touched."

Routine responsibility: every routine that audits or updates a page's content must bump data-updated to the current YYYY-MM as a step in that run. This applies to guide cribs (guide-content-audit, guide-drift-audit, guide-audit-and-fix), to non-guide cribs (vaccines-audit, currency-guide-refresh, stats-pages-monthly-refresh, etc.), and to any session that reviews content as part of its task. Routines that only fix format/validators may skip the bump.

Seed baseline: on the 2026-06-21 migration all 155 live guides were seeded data-updated="2026-06". On 2026-07-22 all non-guide content pages (Trip-Essentials, Best-Of, maps, Guides-Index) were seeded data-updated="2026-07". From here, each page's date advances only when a routine reviews its content.

Validator: TB-11 hard-fails any guide whose toolbar-mount lacks a well-formed data-updated="YYYY-MM". For non-guide pages, brain_check.py warns when data-updated is absent or stale (>12 months without an update). The guide build stub and new-page templates set the attribute to the current month at creation time.

§ 11. Per-guide "🗺 All Stops Map" pill — auto-injected (added 2026-07-14)

What it is: every guide ships a companion page Guides/{City}/{slug}-stops-map.html ({slug} = the guide filename with its _vN version suffix stripped, e.g. Lisbon's guide → lisbon-stops-map.html) — a full-screen self-hosted Leaflet map plotting every stop in the itinerary, with per-day filter buttons. It reuses the vendored Leaflet under Trip-Essentials/Maps/vendor/ (same engine as the unified World Map) and loads web-travel-style.css + mobile.css (it is a map page, not a guide, so it does not load guide-style.css). Its toolbar-mount carries data-depth="2" and, like the Read About companion, no data-updated. Reference implementation: Travel-Website/Guides/Lisbon/lisbon-stops-map.html.

How the pill appears: toolbar.js's injectStopsMapPill() runs on every real guide. It derives the slug from the current filename, fires a HEAD request to ./{slug}-stops-map.html, and — only on a 2xx response — appends a 🗺 All Stops Map pill (class="overview-extra-link") at the end of the .gel Trip Overview extras row, after all static pills (including ✨ Claude Inspiration). It is a no-op when the map file is absent (404), and it skips injection if the guide's static HTML already contains an a[href$="-stops-map.html"]. Because the pill is injected at runtime by the shared script, no guide HTML needs editing — the pill surfaces automatically for any guide the moment its stops-map file is generated.

Cascade note: this is a per-guide companion page (like Read About), not a global toolbar tab — it is NOT in the ITEMS array, Website-Main-Pages-Links.html, or the search_index.json pages list, and it is outside the guide prev/next chain. The canonical placement of the pill within the Trip Overview extras row is owned by CORE RULES Trip Overview.html; this section documents only the toolbar.js injector behavior. As of 2026-07-21, all 224 shipped guides carry a stops-map — the injector is fleet-wide and the pill appears automatically on every guide.

§ 16. Visited pill — runtime-injected, personal (added 2026-07-19)

What it is: when a guide belongs to a city the owner has been to, toolbar.js injects a gold-outline "Visited" pill at the very bottom of the page — after the #nearby-guides section (the last section on every guide), right-aligned. The pill is personal: it reads localStorage key tve-visited-{slug} (where {slug} is the guide filename with its _vN suffix stripped — e.g. lisbon_v4.html → tve-visited-lisbon). Other visitors never see it because they have no matching localStorage entry on their device.

How it works: _injectVisitedPill() runs on every isRealGuide page at DOMContentLoaded. It derives the slug, checks localStorage.getItem('tve-visited-' + slug), and — if set — creates a right-aligned <div> wrapper after #nearby-guides (fallback: last .extras-section) containing a <span id="tve-visited-pill"> styled inline: background:#fff; border:1px solid #c8a44a; color:#8a6c1a; border-radius:6px; padding:6px 16px; font-size:13px; font-weight:500 — matching the "Also on this site" pill visual language. Text reads "Visited". Clicking the pill removes the localStorage key and removes the wrapper (unmark). If the key is absent the function is a pure no-op.

Setting the visited flag — automatic sync from Guides-Index: Guides-Index.html runs a sync script on every page load that loops all a.dest-card[data-status] elements: cards with data-status="been" call localStorage.setItem('tve-visited-{slug}', '1'); cards with any other status call localStorage.removeItem('tve-visited-{slug}'). This keeps localStorage perfectly mirrored to the index — opening the Guides Index page once is all that is needed before the pill appears on any visited guide. The sync uses the same slug-extraction regex as _injectVisitedPill() (/^(.+?)(?:_v\d+)?\.html$/) applied to the card's href basename.

What does NOT change in guide HTML: guides ship exactly as they always have. No new HTML is added to guide files for this feature. The data-status="been"/"want" attributes on Guides-Index.html .dest-card elements are preserved (inert for filtering now, but reserved for the Step 4 world-tracker integration).

§ 19. Alternative hotel recommendations — HOTEL_ALT_DATA (updated 2026-07-24)

§ 20. Sticky stop-name strip — guide pages only (added 2026-07-23)

What it is: a slim fixed strip that appears at the top of the viewport while the reader is mid-way through a stop on a guide page. It shows the stop number and name — 📍 3. Kunsthaus Zürich — so context is never lost on long 6–8 stop days where the stop header has scrolled off screen by the time the reader reaches restaurant picks or a transport note.

How it works: _injectStopStrip() runs on every isRealGuide page. It creates a #tve-stop-strip div (position:fixed; top:0; left:0; right:0; height:28px; z-index:101; pointer-events:none) and appends it to <body>. An IntersectionObserver (threshold 0) watches every .stop-header element. When a header's boundingClientRect.top < 0 and it is no longer intersecting, it is added to an aboveViewport Set. The update function finds the last header in DOM order that is in that Set and writes its .stop-num + .stop-name text to the strip. The strip is hidden (display:none) when the Set is empty — i.e. the reader is above the first stop, or has scrolled back to the top.

Strip styling: background:#f0ede5; border-bottom:1px solid #ddd8cc; font-size:12px; font-weight:500; color:#6b6860 — matching the guide's warm muted palette. pointer-events:none means it never intercepts taps or clicks. z-index:101 keeps it below the dropdown menus (z-index:1000) and the mobile hamburger overlay (z-index:1001–1002) while sitting above all guide content.

Why fixed, not sticky: the toolbar itself is position:relative and scrolls away. A position:fixed; top:0 strip therefore only appears over the toolbar while both are on screen — but by the time any stop header exits the viewport (the first stop begins well below the ~60px toolbar), the toolbar has already scrolled off. There is no visual overlap in practice.

Zero guide HTML changes: the strip is fully runtime-injected. No attribute, class, or markup change is needed in any guide. Selectors used: .stop-header, .stop-num, .stop-name — all guaranteed present by Guide Structure.html and the stub generator.

Guard: silently no-ops if window.IntersectionObserver is absent (very old browsers), or if the page has no .stop-header elements.
