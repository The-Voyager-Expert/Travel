📌 Purpose

Read this file before touching any CORE RULES file, validator, section name, icon, guide CSS, or starting a new guide build. It lists every downstream file that must follow each type of change. Nothing should ship without checking its cascade first.

Direction is one-way. Changes trigger cascades; this map documents the cascades. The CORE RULES HTML files are the source of truth — this map never overrides them.

📄 CORE RULES file edited

Any change to a file in Brain/CORE RULES/ — wording, thresholds, format rules, section additions.

⚠️ Cascade — CORE RULES file edited
Affects: validator integrity check · downstream rule consumers · both Brain docs
  • Regenerate checksums — run Brain/scripts/update_core_rules_checksums.py to update Brain/scripts/core_rules_checksums.json
  • Run Brain/scripts/doc_workshop_validator.py — confirms all CORE RULES HTML files still conform to Universal Formatting Rules; fix any violations before closing the task
  • Check if Brain/scripts/validate_itinerary.py references the changed rule — if yes, update the validator logic and label to match
  • Update Brain/Reference/Validator Index.html if any validator check was added, changed, or removed
  • Update Brain/Reference/Rule Dependencies.html — update the relevant icon card, concept block, or threshold entry to reflect the new rule value
  • Update Travel/CLAUDE.md — if the change introduces, modifies, or retires a behavioral rule or tripwire, add or update the matching DriftyCat entry; if the change affects the build-phase list or session ritual, update those sections too
  • Check Brain/Reference/Separation Map.md — if the change creates, moves, or removes rule ownership for a concept, update the relevant row
  • If the change affects a threshold or format used in multiple CORE RULES files, check Rule Dependencies.html for all files that share that rule — update them all before regenerating checksums
  • CORE RULES files are read-only and can only be changed by explicit request and permission. Do not edit without that gate.

⚠️ Cascade — validator check added or changed
Affects: coverage doc · dependency map · all guides (run to confirm)
  • Update Brain/Reference/Validator Index.html — add new check under the correct section; mark ✅ / ❌ / ⚠️; update section item count in the section heading
  • Update Brain/Reference/Rule Dependencies.html — if the check enforces a rule that is cross-referenced, update the relevant icon card or concept entry

  • Add a matching entry to the embedded CHANGELOG list at the top of Brain/scripts/validate_itinerary.py — its own header states "ANY addition, removal, or edit to a check = one entry here", and the validator prints it on every run. Two changelogs exist; update both.
  • Run Brain/scripts/audit_all_guides.py (or validate each guide individually) to confirm no regressions across all guides
  • If the check enforces a new format rule not yet documented in CORE RULES, log it in 🔧 Rules for Update in To_Do_List.md before shipping

🎨 CSS rule changed

Any change to the guide stylesheet (Travel-Website/assets/guide_v3.css) — new rule, retired rule, value change.

⚡ Cascade — CSS rule changed
Affects: all guides (visual) · validator drift sentinels if a class is retired
  • Edit Brain/Reference/Guide Style.css (source of truth), then run python3 Brain/scripts/guide_tools.py sync-css to copy it to Travel-Website/assets/guide_v3.css — never edit guide_v3.css directly. (guide_v2.css and guide_v2_pre_travel_essential_palette.css are retained previous formats — never deleted, never written to.)
  • If the change retires an old class or id, add a drift sentinel to Brain/scripts/validate_itinerary.py that hard-fails any guide still using the old class — add a matching entry to validate_itinerary.py's embedded CHANGELOG; update Brain/Reference/Validator Index.html to log the new sentinel check and add a changelog entry to its .meta block
  • Run Brain/scripts/audit_all_guides.py across all guides to confirm no visual regressions and no new drift-sentinel failures
  • If any link color or CSS custom property that controls link color changes (e.g. --c-link, --c-title-bg, per-theme selector colors), update CANONICAL_LINK_BLUE and LINK_COLOR_ALLOWLIST in Brain/scripts/validate_itinerary.py and add a changelog entry — this has caused silent drift twice (2026-05-31, 2026-06-01)
  • Inline style= attributes with layout properties in any guide are a validator hard-fail — do not use inline styles to work around a CSS change
  • "Updated Month Year" stamp does NOT update on CSS or toolbar changes. The stamp reads the guide HTML file's own mtime — format-only changes (guide_v3.css, toolbar.js) never touch guide HTML, so the stamp stays at the prior content-edit date. This is intentional: "Updated" means content currency, not format currency. Never touch guide HTML files solely to bump the stamp after a CSS change. (Full spec: Brain/Reference/Toolbar.html § 10.)
