§ 2. Prev / next arrows

Guide pages that belong to a sequence carry data-prev and data-next attributes on the toolbar mount div. When present, the script renders fixed left and right side arrows pointing to the neighbouring guide. Both are optional — a guide with no predecessor omits data-prev; a guide with no successor omits data-next.

<div id="toolbar-mount" data-depth="2" data-maxwidth="940" data-prev="../CityA/guide_vN.html" data-next="../CityC/guide_vN.html"></div>

Paths are relative to the guide file's own directory. Each value points to the neighbouring guide's HTML file — not to a folder or an index page. Trip Essentials pages do not use these attributes.

§ 3. Chain integrity

The guide sequence is a doubly-linked chain: every guide's data-next matches the following guide's data-prev, and vice versa. A mismatched path, a stale filename after a version bump, or a guide added without updating its neighbours leaves one or both arrows dead. When a guide is added or renamed, the two adjacent guides in the chain are updated in the same change.

§ 4. Canonical order and guides_index

The canonical guide order is defined in Travel-Website/Guides/guides_index.html. Every guide card in the index carries matching data-guide-prev and data-guide-next attributes. When the chain changes, both the guide file's mount div and the index card are updated together.

The index groups guides by country (one section per country, countries ordered alphabetically — split from the former region grouping 2026-06-07). Within each country section, guides are ordered alphabetically by city name — case-insensitive, diacritics normalized (Ålesund sorts as Alesund, Montréal as Montreal). The validator enforces this order and reports the first out-of-order city per country section. The carousel chain (data-guide-prev/data-guide-next) ignores the grouping and is strictly global A→Z.

§ 5. Adding a new guide to the chain

Insert the new guide between its two neighbours. All five steps happen in a single pass — partial updates break the chain:

  1. Add the new guide's card to guides_index.html in the correct country section, placed in alphabetical order by city name within that country, with the correct data-guide-num attribute (sequential number in the index's document order). When the country has no section yet, create one in alphabetical position using the existing section pattern (flag + country name + count).
  2. Update the predecessor card's data-guide-next and the successor card's data-guide-prev to point to the new guide.
  3. Update the banner counts (guides + countries) and the country section's guide count in guides_index.html.
  4. Set the new guide file's own toolbar-mount data-prev and data-next to match its neighbours.
  5. Add a pin for the city to the matching region map in Travel-Website/Trip-Essentials/Maps/ — Travel-Website/Trip-Essentials/Maps/Europe-Map.html, Travel-Website/Trip-Essentials/Maps/US-Map.html (North America), Travel-Website/Trip-Essentials/Maps/Asia-Map.html, Travel-Website/Trip-Essentials/Maps/Africa-Map.html, Travel-Website/Trip-Essentials/Maps/Oceania-Map.html, Travel-Website/Trip-Essentials/Maps/South-America-Map.html, or Travel-Website/Trip-Essentials/Maps/Caribbean-Map.html. Add an entry to the PINS array with ['CityName', lon, lat, '../../Guides/City/file.html']. The ship gate scans all seven maps — the guide will not ship without a matching pin in one of them.

§ 6. Scroll progress bar

The script renders a thin progress bar fixed along the top of the viewport. Its width tracks how far the reader has scrolled through the page. It is passive — no markup or configuration per page.

§ 7. Scroll buttons

The script renders two persistent buttons fixed at the right edge of the viewport, vertically centered, on every page. The upper button carries an up chevron (∧) and scrolls to the top of the page; the lower button carries a down chevron (∨) and scrolls to the bottom. Both are always visible. The buttons match the prev/next arrow style — same background, border, and dimensions — with SVG chevron paths in place of text characters.

§ 8. Guides index content search

The guides_index.html search box (#guide-search) does two things at once. Its original job — filtering the city cards by name/country — is unchanged. Added 2026-06-14, a second, additive module also performs full-text content search across every guide: as you type (2+ characters), a dropdown panel below the box lists in-guide matches — stops, attractions, tours, restaurants, and section names — each deep-linking straight to the relevant guide and section (e.g. Barcelona/barcelona_v1.html#day2). Results group by city, rank best-match first, highlight the matched text, and fold diacritics (so sacre finds Sacré-Cœur). The module is suppressed in the flight-time view and never touches the card filter or the flight parser.

It reads Travel-Website/assets/search_index.json, lazy-loaded on first keystroke. That file is generated by Brain/scripts/build_search_index.py, which parses every guide's HTML for searchable entries. The index rebuilds automatically as the last step of guide_tools.py ship (see Brain/Reference/Ship Checklist.html), so a freshly shipped guide is searchable immediately. Run the script by hand only after editing a guide without shipping.

Search behaviour on the Trip Essentials pages — including how a no-match query collapses the page and hides the shared footnote — lives in Travel-Website/Trip-Essentials/Essentials-Pages-Rules.md.
