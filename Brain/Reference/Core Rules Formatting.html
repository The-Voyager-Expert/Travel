§ 3. Building blocks

Banner — the read-only notice (top + bottom of every CORE RULES file)

A light-blue box with red text. It opens and closes every CORE RULES file. The .banner class supplies the box; the content is plain lines separated by <br>, with the ATTENTION line bold via <strong>.

Style (.banner in Core Rules Style.css): red text #cc0000 on light-blue #e8f4fd, padding 10px 14px, border-radius 4px, font-size 16px, line-height 1.5. No uppercase, no letter-spacing — only the ATTENTION line is bold.

Exact HTML — copy verbatim:

<p class="banner"><strong>ATTENTION: THIS DOCUMENT IS READ-ONLY AND CAN ONLY BE EDITED BY REQUEST</strong><br> Read the formatting rules first — Brain/Reference/.<br> CSS, HTML, and validator code are not permitted in rule text.<br> Rules are written like a legal document — active voice, generic phrasing, no specific examples. The rules state what is; the validator catches what isn't.</p>

Adding a line: insert the new sentence as its own <br>-terminated line inside the same <p class="banner"> — nothing else changes. Apply the identical edit to every CORE RULES file's top and bottom banner.

Title + subtitle

<h1>{emoji} {Doc Title}</h1> <p class="subtitle">Source of truth — <code>Brain/CORE RULES/{filename}.html</code></p>

Meta block (Status + Summary)

Always wrapped in <div class="meta"> — gives the bottom border separator.

<div class="meta"> <p><strong>Status:</strong> UNIVERSAL — always ships.</p> <p><strong>Summary:</strong> {description}</p> </div>

Section headers

<h2>§ N. {Title}</h2> ← top-level section <h3>{Subtitle}</h3> ← sub-section within a section

Template block

For format examples — what to write, what a row looks like. Monospace, gray background, left border. Preserves line breaks exactly.

<div class="template">{Venue Name} ← linked to site 🏛️ Opens {time} 📍 {Address} 🚶 {N} min walk</div>

Callout box

For rules and reminders — things that need to stand out but aren't a code template.

<div class="callout"> <p>Row order is locked: <em>info → map → walk</em>.</p> <p>Walk is always last.</p> </div>

Inline code

<code>{inline snippet or path}</code>

Lists

<ul> <li><strong>{Label}</strong> — {description}</li> </ul> <ol> <li><strong>{Step}</strong> — {description}</li> </ol>

§ 4. Source whitespace

Keep the source clean — it's easy to read and easy to diff.

  • One blank line before each <h2>. No blank lines between paragraphs within a section.
  • Indent list items and <div class="meta"> / <div class="callout"> children with 2 spaces.
  • No trailing blank lines at the end of <body>.

§ 6. Guide HTML typography

Guide HTML files reference the shared stylesheet via a <link> tag. The font is Roboto (Google Font), loaded via @import inside guide_v3.css — no separate <link> tag for the font in guide HTML files. The stylesheet link goes in <head> and uses the canonical relative path:

<link rel="stylesheet" href="../../assets/guide_v3.css">

The path is exactly ../../assets/guide_v3.css (two levels up from the guide HTML to the site root, then into assets/). The deployed CSS lives at assets/guide_v3.css for GitHub Pages — that copy is what every guide reads from, both locally and on Pages. The source of truth is Brain/Reference/Guide Style.css — whenever it's updated, copy it to assets/guide_v3.css too. Per Rules for Claude.html § 4.

Never use <link rel="stylesheet" href="../../Brain/guide_v3.css">. That path works locally but breaks on GitHub Pages because Brain/ is not deployed. The validator hard-fails on this exact path.

The validator hard-fails on a missing or wrong stylesheet link and on the forbidden ../../Brain/guide_v3.css path. Google Fonts is loaded via @import inside guide_v3.css — a separate <link rel="stylesheet" href="fonts.googleapis.com/..."> tag in the guide HTML is still not permitted; the font import belongs in the CSS file only.

No commas in titles or row data

In headings, names, and the data on entry rows — ratings, hours, addresses, and the like — do not use a comma. Separate items with the middle dot · instead. Normal commas inside a written description sentence are fine.

No parentheses anywhere

Parentheses ( ) are never used in guide visible text — not in titles, not in row data, not in description prose. Put the information after a middle dot · instead.

Separators — two jobs

  • Middle dot · (a space on each side) separates two different items on a row — 4.6⭐ · 200+ reviews.
  • Hyphen - (a space on each side) joins the two ends of a range — 9:00am - 5:00pm, Monday - Wednesday.

In description prose, either a middle dot · or a hyphen - may be used as a separator. No en-dash –.

Title Case for names

Venue, restaurant, café, tour, station, and dish names use Title Case — capitalize each significant word (The Coffee Collective, Museum of Modern Art). Short connecting words — a, an, the, of, and, in, on — are always lowercase. Never ALL CAPS.

§ 7. Guide extra-section entry HTML structure

This section governs the HTML wrappers inside extra sections that ship multi-entry lists with a name, an optional description, and a data box (Cappuccino, Restaurants Near Hotel, Downtown, Michelin). The content rule (what the entry contains, what counts as a qualifying entry, length caps) lives in each section's CORE RULES file — this section only governs the HTML tags that wrap that content.

Every entry follows this exact three-element sequence:

<div class="extras-sub">{Name}</div> <p>{description text — content rule sets the char cap per section}</p> <div class="{section}-box">…</div>

Element rules

  1. Name heading. Always <div class="extras-sub">. Plain text inside (no link, no emoji prefix). The name text format is governed by each section's CORE RULES file.
  2. Description. Wraps in <p> — never a bare <div> and never a <div> with a class. Reasons: (a) <p> is the semantic element for prose text; (b) it gives the validator a single, unambiguous target to enforce per-section char caps against; (c) it lets CSS style descriptions distinctly from box rows. The description element appears immediately after the .extras-sub heading and immediately before the data box. Optional — omit the entire <p> element when an entry has no description (no empty <p></p>).
  3. Data box. Wraps in a section-specific class (.entry-body, .shows-box, .transit-box, .station-box, per guide_v3.css). The box always follows the description (or the heading, when there is no description).

What is not allowed between heading and box

  • No bare <div> elements carrying description text — the validator now treats those as a hard fail and they will block ship.
  • No multiple <p> elements per entry — one description paragraph, then the box.
  • No additional headings (h2/h3) inside an entry.

Validator enforcement

The shipping validator (Brain/scripts/validate_itinerary.py) enforces this structure for every extras section that carries entries with descriptions:

  • Per-section description-length check — fires per section's CORE RULES cap (Downtown ≤80, Restaurants Near Hotel ≤80, Michelin ≤80, Cappuccino's cap in its own file). Anchored on each .extras-sub heading; reads the immediately-following <p> as the description.
  • Bare-div description check — fires when a plain <div> (no class) appears between an .extras-sub heading and the section's data box. This is the failure mode that previously let over-length descriptions slip through (the validator saw only <p> elements; bare <div> wrappers carried unchecked content past the cap).

Review-link heading (Cappuccino · Restaurants Near Hotel · Downtown)

Venue sections that carry a rating use this heading row: the venue name as plain text, a middle-dot separator, then the rating and review count — [Venue] · N.N⭐ · NNN+ reviews. The name is never a link. The N.N⭐ · NNN+ reviews is the blue link — the only clickable element on the row — a blue, no-underline anchor (<a class="review-link">) to the venue's Yelp, Google Reviews, or Local Guide page. What is a link is blue; everything else on the row is plain.

Tours is the related-but-separate case: there the tour name is the blue link (to the operator's product page), not the rating. Each section's CORE RULES file states which element links.

§ 8. CORE RULES entry format — locked pattern

Every CORE RULES file that shows an entry format uses the same four elements. This is the only way — no variations, no exceptions.

The four elements

1. Placeholder (input slot) — <code>[Title Case]</code>

Grey background chip. Shows what the guide builder needs to fill in — "this is where you put the real value." Title Case always — every word capitalized. No domain extensions ([Viator] not [Viator.com]). No curly braces, no parentheses wrapping placeholders.

Two placeholders on one row: <code>[A]</code> <code>[B]</code> — space between, no parens around them.

Static words around a placeholder stay plain — only the variable gets <code>:

Search day trips from <code>[City]</code> across reputable travel guides…

2. Hint text — <span class="note">…</span>

Grey text, no background. For char limits and clarifications only — never for placeholders.

<span class="note">≤320 chars — what the place is and why it's worth the trip</span>

3. Link marker — <span class="link">[Name]</span>

Blue bold. No underline, not clickable — a <span>, never <a href>. Shows where a clickable element goes in the real guide. Name = the service or site name, Title Case, no domain extension: [Viator], [Booking Site].

4. Plain text

Everything else — rule prose, static words. No <em>, no <strong>, no extra tags.

Address row — locked pattern

Every entry that has a location uses this exact pattern for the 📍 row. The placeholder name uses the entry type as a prefix ([Restaurant Address · Neighborhood], [Cafe Address · Neighborhood], [Station Address · Neighborhood], etc.). The hint note always cites Links.html as the authority for the format. Cross-references name the file only — never the § number inside it (Dani convention, 2026-06-06); internal references within the same file may use § numbers.

📍 <span class="link">[Type Address · Neighborhood]</span> <span class="note">← per Links.html</span>

The word "Type" is replaced by the specific noun for that section. The format [Type Address · Neighborhood] shows the two-part shape: street address first, neighborhood second, separated by · . No city (unless the address is outside the guide's destination city), no postal code, no country.

The yellow box

Entry tables use <table class="entry">. The yellow background (#fef9e0) means: "this is the format to produce." It is the output template — what the guide builder writes in the actual guide. Not a rule description, not a note — the exact pattern to replicate.

Font sizes

All elements in entry tables must be the same font size — 16px (body). The universal CSS sets <code> to 13px; the local style override corrects this:

code { font-size: inherit; }

<span class="note"> and <span class="link"> inherit 16px naturally. Any prior font-size values in existing files are wrong — ignore them, apply the local override only.

No italic

The universal CSS does not define italic. Any <em> tag in a CORE RULES file was manually added and is wrong — strip it on sight. Sole carve-out: the five format-exception files (Rules for Claude.html) use <em> intentionally to mark illustrative phrases; do not strip it there.

Required local style block — every CORE RULES file

<style> code { font-size: inherit; } .entry { background: #fef9e0; } </style>

Add li { margin-bottom: 12px; } only when the file has <ul> or <ol> lists.

Description / summary row

An icon-led row whose placeholder holds the entry's free-text description, followed by a hint note giving the character cap and a short phrase naming what the text should cover. The format is identical everywhere — only the icon, the cap, and the "what it covers" phrase change per section.

{icon} <code>[Description]</code> <span class="note">≤N chars — {what the text covers}</span>

The cap is a maximum, written ≤N chars. The phrase after the em-dash names what to put in — Day Trips ↳ ≤320 "what the place is and why it's worth the trip", Tours ↳ ≤320 "route, stops covered, and the kind of experience". Per-section caps are listed in Brain/Reference/Guide Entry Counts.html.

Example — Day Trips entry format

<table class="entry"> <tr><td><code>[Destination]</code> <code>[Travel Time]</code></td></tr> <tr><td>↳ <code>[Description]</code> <span class="note">≤320 chars — what the place is and why it's worth the trip</span></td></tr> <tr><td>🎫 book at: <span class="link">[Booking Site]</span></td></tr> </table>

Mandatory last two sections — every Extra Section file

Entry Format(s) and When none qualify are always the last two sections, in this order, no exceptions:

§ N. Entry Format(s) <table class="entry">…</table> § N+1. When none qualify Ship the negative-finding line: <table class="entry"> <tr><td>No qualifying [items] from <code>[City]</code>.</td></tr> </table>

§ 10. Reserved placeholder names

These placeholder tokens have fixed meanings across all CORE RULES files and guide builds. Never repurpose them for a different slot.

  • [City] — the city of destination. The city the guide is built for.
  • [Hotel City] — the hotel for the guide city. Used in section rules to identify when a restaurant or café is inside the guide hotel. Example: for a Cannes guide, [Hotel City] resolves to Hôtel Martinez.
  • [Type Address · Neighborhood] — the location address for an entry. "Type" is replaced by the specific noun for the section (Restaurant, Cafe, Hotel, Station, Venue, Meeting Point). Format: street address and neighborhood separated by · . No city, no postal code, no country. Per Links.html.
