🔗 Links

Phase: 1 — Technical prerequisites (also applies during Phase 4 per-stop and Phase 5 per-section)

Summary: Verification gates and format conventions for every link in the guide — Wikipedia, Google Maps, bot-blocked booking URLs, and the target="_blank" rule.

§ 1. Live verification

Every link is verified before ship — every time, including every edit inside a session. A one-character URL change re-runs verification. Two checks must pass:

  • Returns HTTP 200.
  • Page carries ≥100 words of editorial content — no booking-stub shells.

A URL failing either check is replaced with a different verified link. When no verified replacement exists, the row is omitted.

§ 2. Subject-drift check

HTTP 200 + ≥100 words of prose is not enough — a URL with the right shape can return a healthy page about the wrong subject (TripAdvisor d-ID reassignment, Wikipedia slug drift).

The third gate fetches the URL, extracts the first <h1>, and matches it against the stop's venue name. Applies to every TripAdvisor Attraction_Review URL and every Wikipedia /wiki/{Slug} URL in the guide.

§ 3. Verification log

Every guide carries Guides/{City}/_build/verification_log.json with one entry per bot-blocked booking URL — Viator product, GetYourGuide product, Michelin restaurant, TripAdvisor attraction.

{
"url": "{full booking URL}",
"result": "PASS",
"verified_on": "{ISO date}",
"method": "site: search snippet inspection"
}
  • Logged PASS → green. A PASS entry permanently clears the warning for that URL — there is no age expiry (a PASS never goes stale).
  • Missing entry or non-PASS → WARN — re-verify before ship.

§ 4. Bot-blocked platforms

The following platforms block direct fetch. Verify via site: search inspection — never direct fetch. Pattern: site:viator.com {city} {attraction}. For the full authoritative list and current status of every bot-blocked platform, see Brain/Reference/Platforms.md — that file governs. Any platform marked ⚡ or ❌ there skips direct fetch entirely.

  • Viator
  • GetYourGuide
  • TheFork
  • Michelin (guide.michelin.com)
  • TripAdvisor

§ 5. 📖 Wikipedia — the only source for the reading row

Every stop's 📖 link goes to the English Wikipedia article for that stop's specific subject. No other source qualifies for the 📖 slot.

URL: https://en.wikipedia.org/wiki/{Article_Slug}

Link text: just Wikipedia — no descriptor, no article-title appended, no "Official Site" tag.

The article must be the stop's specific subject — the actual venue, monument, neighborhood, or attraction. Content gate: ≥100 words of article prose.

Every stop carries a 📖 row. A missing 📖 row is a hard failure. A guide may omit the 📖 on at most one stop, and only when no substantive English Wikipedia article exists for that subject and a full written explanation is logged in that guide's _build/verification_log.json as an entry with platform: wikipedia-omit (or result: OMIT) and a substantive notes field of at least 40 characters. Two or more stops without a 📖 row fail regardless of the log. A free-standing inline <!-- no-wikipedia: … --> comment no longer exempts a stop on its own — the explanation must live in the guide log.

§ 6. 📍 Google Maps — the address is the link

Every address in the guide is a clickable Google Maps link — every time. The address text itself is the anchor; there is no separate "Google Maps" or "Apple Maps" row.

URL pattern:

https://www.google.com/maps/search/?api=1&query={url-encoded-full-address}

The address should provide the street name and neighborhood. The city is included only when the address is in a city different from the destination [City]. No postal code. The country is included only when the address is in a different country from the destination [City].

This applies to every address in the guide — title-card hotel, ticket-box venues, station entries, restaurant entries, Michelin entries. Same convention everywhere.

§ 7. External links open in a new tab

Every external link in the guide carries target="_blank".

§ 8. No underline on any link

No link in a shipped guide carries an underline. The shared stylesheet removes underlines globally; a guide must not reintroduce one on any element, inline or otherwise. A color change is the only visual marker distinguishing a clickable element from plain text.

§ 9. Internal links encode spaces — never a raw space in a path

A folder or file name may contain a space. A link that points to a local guide or page file must not. Every internal link encodes each space in its path using percent-encoding; a raw space is never written inside the path of an internal link.

This governs every link to a local file — navigation links, predecessor and successor references, map-pin link entries, and the links written into the shared index and data pages. It does not govern links to an external service that carries an address or search term in a query string; those follow § 6.
