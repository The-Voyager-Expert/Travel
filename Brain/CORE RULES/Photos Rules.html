📸 Photos Rules

Phase: 1 — Technical prerequisites (applies during Phase 4 per-stop build)

Summary: Wikimedia Commons sourcing, license bar, harvest recipe, and placement for every stop photo. One photo per stop, real and live-verified.

§ 1. Source

Every photo is a real photo of the stop's subject. No placeholders — no Picsum, no placehold.co, no Unsplash seed URLs. Wikimedia Commons is the default source.

§ 2. Acceptable licenses

  • Public Domain
  • CC0
  • CC-BY
  • CC-BY-SA

Non-commercial-only and unclear licenses are excluded.

§ 3. One photo per stop

Each stop gets one photo genuinely depicting the stop's subject — not a generic city shot. When multiple strong candidates exist, apply the criteria below by stop type.

§ 3a. Photo selection criteria by stop type

Three cases. Apply the first one that matches:

  • The structure is the attraction — the building's form or design is the explicit reason to visit. Exterior or facade shot is correct. Applies only when the stop exists because of what the building looks like — not because of what it contains or what happens inside. Examples: Eiffel Tower, Hallgrímskirkja, Harpa. A museum, gallery, or theater is never in this category even if the building is architecturally notable.
  • The interior is the attraction — museums, galleries, theaters, and any venue where the content inside is the draw. Interior shot showing exhibits, artwork, collections, or performance space is correct. A building exterior is a last resort, used only when no interior shot with acceptable license exists.
  • Everything else — show the subject itself. For nature: the natural feature (never a visitor centre, entrance sign, car park, or building). For squares and plazas: the space, not the buildings around it. For neighborhoods and districts: the defining scene. For viewpoints: the view. A building exterior never wins in this category.

§ 4. Harvest during research

Harvest each stop's photo in the same pass as researching the stop — not as a separate cleanup phase at the end of the build. Stop research and photo resolution belong on the same pull, so a stop never lands in the guide with the photo deferred to "later."

§ 5. Harvest recipe

⚠️ web_fetch is blocked for commons.wikimedia.org in Cowork. Never use web_fetch to open a Commons URL directly — it returns "Web fetch was not allowed." Use WebSearch (site: searches) to find filenames, then commons_photo.py via bash to resolve the 800px URL. This is the only working path.

Wikipedia article images and Commons are separate sources. Wikipedia editors curate one main image per article; Commons hosts thousands of public-domain and CC-licensed photos that don't appear on the Wikipedia page. A Wikipedia article with no main image tells you nothing about what Commons has — always run the Commons search.

Per stop:

  1. Commons category search first — WebSearch for site:commons.wikimedia.org/wiki/Category: {Subject}. Inspect result snippets for filenames. Do not web_fetch the category URL.
  2. Commons direct search as a secondary path: site:commons.wikimedia.org {subject} {city} via WebSearch.
  3. Wikipedia article images as a tertiary check only — the article's main image often appears in Commons too, but Commons usually has stronger candidates.
  4. Pick one photo that clearly depicts the subject — apply § 3a stop-type criteria to decide between exterior, interior, or scene.
  5. Confirm subject match + license from the search snippet or the filename.
  6. Resolve the 800px URL via bash: python3 Brain/scripts/commons_photo.py "File:{Filename}.jpg". This calls the Commons API and returns the canonical thumbnail URL. Never construct the upload.wikimedia.org hash path manually.
  7. Download — the only path (hotlinking is banned): always download the image to Guides/{City}/_build/assets/ and ship it from there as a relative path. Run python3 Brain/scripts/commons_photo.py --download "Guides/{City}/_build/assets/800px-{Filename}.jpg" "File:{Filename}.jpg" — this fetches the original file and resizes it to 800px, working around the sandbox CDN restriction (direct thumbnail downloads from upload.wikimedia.org return HTTP 400, but --download pulls the original instead). Reference as <img src="_build/assets/800px-Foo.jpg" alt="…">. Never ship an <img src> pointing at upload.wikimedia.org — the validator hard-fails every hotlinked Commons URL, and the old <!-- hotlink: … --> comment no longer authorises one. A genuine resolution failure is a "No pictures found" result, never a hotlink.

§ 6. Technical failures — rate limits, API errors

A rate limit or API error is NOT a "No pictures found" result. They are different things. When commons_photo.py or a WebSearch call fails with a rate limit or timeout:

  1. Wait 10–15 seconds and retry the same call once.
  2. If still failing, try the alternative path: if commons_photo.py failed, try site:commons.wikimedia.org File:{filename} WebSearch to get the URL from the snippet instead.
  3. If all paths fail, note the stop name and the filename already found in Guides/{City}/_build/build_state.md, keep building the rest of the guide, and retry the photo before the ship gate. Do not write "No pictures found" for a rate-limited stop.

"No pictures found" is reserved for when the search genuinely finds no suitable licensed photo for the subject — not for technical failures.

§ 7. When none qualify

After the full Commons category + direct search comes up empty (no suitable licensed photo exists for the subject), ship a visible "No pictures found" line in place of the photo block:

<div class="stop-photos stop-photos-empty"><em>No pictures found.</em></div>

Guide-level cap — at most 1 "No pictures found." stop per guide. A single genuinely empty stop is acceptable. Two or more signals the harvest was skipped or abandoned — a process violation. The validator hard-fails any guide with 2 or more empty photo wrappers regardless of whether each wrapper is individually valid.

§ 8. Alt text

Short, plain. Just the subject name. No "photo of" prefix.

§ 9. Storage

Photos always download to Guides/{City}/_build/assets/ and ship from there as relative paths — this is the only permitted storage path. Use commons_photo.py --download (§ 5 step 7), which fetches the original and resizes to 800px, bypassing the sandbox CDN restriction. Hotlinking to upload.wikimedia.org is banned: the validator hard-fails every Commons hotlink, and the retired <!-- hotlink: … --> comment no longer authorises one.

§ 10. Weak-filename patterns

Certain filename fragments reliably indicate a generic exterior shot rather than the most compelling depiction of the subject. These patterns are treated as a quality signal — not a subject-type rule: Building, Main_Building, Exterior, Facade, Outside, Front_view. A photo whose filename contains one of these patterns is presumed weak and must be overridden by a better candidate unless no better-licensed alternative exists.

This applies even to architectural landmarks (§ 3a), where a facade shot is the correct subject choice. The filename flag means: look for a stronger, better-framed facade shot first. When the weak-filename photo is genuinely the best available option, use it and add the sign-off comment — the intent is intentional confirmation, not a block.

When no alternative exists, a sign-off comment must precede the stop's photo block confirming the choice is intentional. The validator enforces this rule.

§ 11. Placement

Photos (or the "No pictures found" line) are always the last element inside the stop block, regardless of stop type. The photo block sits directly under the stop block — not nested inside a ticket box.

Sign-off comment shape when a weak-filename photo is intentional (no better alternative exists):

<!-- photo-ok: no better-licensed alternative found for {stop name} -->
