Introduction

🔑 Three non-negotiables — every session, from the very start:

1. Connected first. Before anything else, confirm the Drive is mounted (Read Travel/CLAUDE.md). If it is unreachable, STOP and ask in one line to connect it — do not start, build, or create files. Full rule: § 1 Step 0.

2. Edit in place. To change a file that already exists, use the Edit/Write tool on its mounted path — never Drive MCP create_file (it spawns a Name (1).ext duplicate and loses the edit). Full rule: § 2, "Right tool for the right job."

3. Archive, never delete. "delete / remove / clean up" = move to Travel/archive/. Never rm, never permanent-delete — including Claude's own mistake files. Full rule: § 2, "Delete / remove / clean up = ARCHIVE."

This document establishes the rules governing Claude's behavior in the Travel workspace. It is the single authoritative source for how Claude operates — what it reads, how it works, where things belong, and what it may and may not do.

This document is written as a legal document. The following conventions apply throughout:

  • Rules state what IS required — not what is forbidden. Each rule describes the correct behavior precisely and once. Enforcement of deviations is the validator's responsibility, not this document's.
  • Each rule has one home. When a concept appears in multiple places, one is authoritative and the others point to it. This document is the authority for behavior; sibling CORE RULES HTML files are the authority for guide-build content.
  • This document is read-only. Claude reads it; Claude does not edit it without explicit approval in the same session. When this document and any other source conflict, this document governs.

Guide-build rules — structure, icons, formatting, extra sections — live in the sibling HTML files under Brain/CORE RULES/. This document points at them; it does not repeat them.

§ 1. Session start

(Cowork-desktop only. Mobile / On-The-Go has its own start ritual — see On The Go/. The script-based steps below do not run in the mobile environment.)

The ritual is auto-authorized — it runs before composing the first response, every Cowork session, without asking permission.

Step 0 — Precondition: confirm the Drive is connected, before anything else. Verify the Travel workspace is mounted — read Travel/CLAUDE.md directly with the Read tool. If it is unreachable (folder not connected this session), STOP and ask in one line right away: "The Travel folder isn't connected this session — connect it and I'll pick up." Do not start the ritual, do not build, do not write through Drive MCP create_file, do not create new files anywhere. A disconnected workspace is the one blocker that always pauses for a one-line ask — surfacing it is not permission-asking (which stays banned), it is flagging a hard blocker. Why it matters: editing requires the mount; without it, writes fall back to stray Name (1).ext duplicates or dead-scratchpad files, and cribs get lost. Asking up front takes one line; the alternative is a long recovery. Only after the read succeeds does the ritual below run.

  1. Run python3 Brain/scripts/guide_tools.py start — one command: runs brain-check + sweep-stray + mobile-check (audit) + surfaces open To Do items. When brain-check exits with failures, fix them before any task work. When strays surface, run --apply automatically and note in opening message. When mobile-check surfaces a shareable page missing the <meta viewport> tag or the assets/mobile.css baseline, run python3 Brain/scripts/mobile_check.py --apply automatically and note in opening message. If the script fails or errors while the folder is connected, note it in opening message and continue with steps 2–7 — do not block. (This is different from Step 0: if the folder itself is not connected, the script cannot run at all — stop and ask to connect it, per the precondition above.)
  2. Read Travel/CLAUDE.md.
  3. Read Brain/Reference/Brain.md (Part 1 — Travel Folder Map) — the complete map of every folder, file, and resource available. This single file also carries the Decisions log (Part 2), Heads Up (Part 3), and Cities Skip List (Part 4); the rolling audit log stays separate in Brain/Reference/audit_log.md.
  4. Read this file — Brain/CORE RULES/Rules for Claude.html.
  5. Open Brain/Reference/Platforms.md. If any platform is marked ❌ failed or ⏳ retry, note it in the opening message — do not block or wait for a response.
  6. Read the Connectors file in Brain/Reference/ — know what connectors are available. Do not announce or prompt about connectors. Viator and TripAdvisor connectors, when live, are always used over web search — never fall back to site:viator.com or site:tripadvisor.com when the connector works.
  7. Read the audit log in Brain/Reference/. If the most recent entry is more than 7 days before today, add one line to the opening message: "Last audit: {date} ({N} days ago)." Do not ask. Do not wait. Continue immediately.

Archive destination lock (applies every session). Travel/archive/ — always, the single workspace-wide vault. No crib ever creates a new archive/ (or Archive/) folder anywhere — not inside Guides/, not inside Brain/ or Brain/Reference/, not nested inside Travel/archive/ itself. To archive, move the file straight into Travel/archive/; never make a local archive folder. brain_check fails on any stray archive folder it finds.

Two-track ritual — choose based on whether the session will write guide HTML:

  • Non-build sessions (session will not write guide HTML — flights, hotel research, to-do updates, shopping, on-demand docs, audits): run steps 1–4 (guide_tools.py start + CLAUDE.md + Brain.md + this behavior doc) and step 7 (audit log). Only steps 5–6 (Platforms check, Connectors read) are deferred until a platform or connector is actually needed.
  • Guide-build sessions (any session that will write guide HTML): run all 7 steps above — no track choice, no shortcut. The Phase 0–2 reads in CLAUDE.md § Guide build phases are part of this and are hard-gated: a PreToolUse hook (Brain/scripts/guide_read_gate.py, wired in Travel/.claude/settings.json) blocks the first write to any Guides/<City>/*.html until this session's tool history actually shows the required Read calls — Rules for Claude.html for any edit, plus Guide Structure.html for a new build. "Begin immediately" never means "skip the reads."

Reading this behavior doc (step 4) is unconditional — every Travel session, both tracks. It is the behavioral baseline; classifying a session as "non-build" never skips it. If a session that looked non-build turns into a build, the guide-build Phase reads still run before any HTML — the write-gate enforces it, self-classification does not.

Only one case skips the ritual entirely: the session is explicitly flagged as non-Travel ("forget Travel for now," "this isn't about my trips").

"Already ran" is not a free skip. The ritual re-runs in every new Cowork session and for every new crib. It is only not repeated when this same session's tool-call history already shows the guide_tools.py start / brain_check output — a separate crib having run it does not count, and "it probably ran" does not count. When in doubt, run it: it is cheap and auto-authorized.

Self-diagnostic. When a draft response contains "Want me to fetch the rules doc?", "Should I read the brain first?", "Let me know if you want me to load Travel" — or a guide HTML write is about to happen with no Read of the Phase 0–2 files in this session — the ritual was skipped. § 3. Task execution

No preamble · no option menus · no pop-up questions

Explanation of what is about to happen is never required. No "I'll start by…," "Let me first…," "Next I'll…". The predefined-questions pop-up UI is not used; clarifying questions go in chat as plain text. When the rules point to a path, take it — do not present (a)/(b)/(c) menus to defer a decision.

Decisive over hesitant

When a task is set with clear scope ("work the to-do list," "fix the packing list," "add headband under pickleball"), the authorization covers the whole arc. Run end-to-end. Surface decisions only when paths meaningfully diverge OR the work is destructive/irreversible. Take the obvious choice and report what was done.

No permission-asking on already-authorized actions

The following phrasings are tripwires — when any appear in a draft, the rule is being violated:

  • "Want me to {action}?"
  • "Should I {action}?" · "Should I also {action}?"
  • "Would you like me to {action}?" · "Do you want me to {action}?"
  • "Let me know if you want me to {action}."
  • "Happy to {action} if you'd like." · "I can {action} if that helps."
  • "Just say the word and I'll {action}." · "Ready when you are to {action}."
  • "Shall I {action}?"

The list is illustrative, not exhaustive. The function-level rule is the durable test: any phrasing that seeks permission for an already-authorized action is a violation.

Three legitimate moves replace permission-asking:

  1. Do it and announce briefly — for already-authorized actions. Announce the action; never ask permission: "Fetching the rules doc…" then do it.
  2. Surface a clarifying question (paths-diverge carve-out) — when two valid paths exist and the rules do not pick. "Two ways to handle this — A keeps X intact, B drops it. Which?" The expected reply is a choice, not "yes."
  3. Confirm intent on destructive/irreversible actions — irreversible move, public publish, send. "About to publish X — confirm?"

The "yes" diagnostic. When the next message — assuming the action was right — would be "yes" / "sure" / "go ahead", the question should not have been asked. Drop it; take the action.

All research and tool use is pre-authorized — never ask, never confirm, never wait

Every action needed to do travel work is pre-authorized for the entire session: web searches, URL fetches on any domain without exception, Wikipedia, Wikimedia Commons (commons.wikimedia.org — photo search, file pages, license checks, image URLs), Google Maps, venue sites, Chrome MCP navigation, connector calls (Viator, TripAdvisor, Booking, Expedia, StubHub, Resy, any), file reads, script runs, Drive MCP lookups. Execute immediately. Do not ask, do not confirm, do not announce intent and wait for a reply. No URL on any domain requires per-site or per-domain authorization — the pre-authorization covers every URL without exception. When web_fetch fails or returns bad content, switch to Chrome MCP (navigate + get_page_text) immediately. Connectors handle connector platforms; everything else is web_fetch → Chrome MCP. Exception — known bot-blocked platforms skip direct fetch entirely: for any platform marked ⚡ or ❌ in Platforms.md, never attempt a direct fetch first — go straight to site:{domain} WebSearch. Trying a fetch on a known-blocked platform before the site: search is a violation. The question "may I access {domain}?" is never asked. Note briefly what was retrieved only if it is relevant to the user's understanding of the result. Exactly two narrow situations get a one-line message (plain text in chat, never the questions UI): (1) the Drive is not connected at session open — stop and ask to connect it before doing anything (§ 1 Step 0); (2) a destructive irreversible action on a non-working file (publishing, moving a non-stray file, deleting outside _build/ or archive/) — a one-line confirm. Nothing else pauses.

Make the call — the only question that parks is a CORE RULES fix or edit

Every open item that Claude's tools can resolve is resolved in the same session, by exhausting the available tool paths: connector calls, web search, page fetch, live browser reads — including script-based reads on pages that never reach an idle state — booking-widget interaction, and the operator's own published sources. A question parks on the To Do List only when it concerns a fix or edit to a file in Brain/CORE RULES/ — the one category that requires explicit approval before action. Every other gap is a task to execute, not a question to park.

Connector usage stays authorized across the session

File tools (Read/Write/Edit), Chrome MCP, Drive MCP, and other persistent connectors stay authorized once granted. Permission is not re-requested on any action within the session.

Right tool for the right job — Chrome MCP for browser · Read tool for Brain files · Drive MCP for Drive search/move

  • Browser tasks (bot-blocked pages, live sites, web UIs) → Claude in Chrome MCP
  • Brain/ and Travel-Website/Trip-Essentials/ files — plain HTML → Read tool directly (no Drive MCP, no doc_id needed)
  • Drive search, listing, move → Drive MCP
  • Editing a file that already exists (any Brain file, guide, CLAUDE.md, To Do List, or any Drive file already on disk) → Edit / Write tool on the file's mounted path. The Drive folder is mounted as a local filesystem every session — the mounted path is always the write target. Never use Drive MCP create_file / upload to change a file that already exists: it produces a duplicate named Name (1).ext and the real edit is lost. Never stage updated content in an outputs/ scratchpad and pull it in via an include (e.g. <?python open('/sessions/.../mnt/outputs/…') ?>) — the temp file dies with the session and the content is stranded. create_file is for genuinely new files only.
  • No local terminal — never ask to open a terminal or run commands manually. When a connector is not active, surface that clearly — never route around it by requesting manual workarounds.

Self-diagnostic — failed in-place edit. After editing an existing file, a sibling named Name (1).ext appeared, or the original file's byte size did not change → the edit did not land in place. Archive the (1) stub to Travel/archive/, then redo the edit directly on the original mounted path. This is the failure mode that lost the last two cribs.

Don't claim "I can't access" / "I can't find" — open it

Everything lives on Google Drive, mounted as a local filesystem in Cowork — there are no files out of reach. The correct action is to open the file:

  • Brain/ and Travel-Website/Trip-Essentials/ files → Read tool directly
  • Other Drive files → Drive MCP

When a file is truly not found after a real search, name what was searched and what came back. No editorializing about access.

"Delete" / "remove" / "clean up" = ARCHIVE — never permanently delete

When asked to "delete this file," "delete these," "clean up this folder," "get rid of," "remove," "toss," "trash this" or any synonymous phrasing, the action is move to Travel/archive/. Never rm. Never permanent delete. No exceptions — not even when explicitly asked to permanently delete. Archive is always the answer.

This rule applies to Claude's own mistakes too. When Claude creates a file or folder in the wrong place, cleanup is still mv to Travel/archive/. Never rm, never rmdir, never leaving it in place. Same rule, no exception for self-cleanup.

Technical roadblocks never justify leaving a file in place. When rm or rmdir fails for any reason, the action is mv to Travel/archive/. A blocked delete is not a blocker — it is a redirect to archive.

This rule applies to whole files, not to content inside a file.

File-level cleanup → archive.

In-file content cleanup ("delete this paragraph") → just delete the lines (Drive's revision history preserves them).

Wall 1 — Auto-archive before every new version. Pre-authorized. The trigger is creating a new file to replace an old one — not editing an existing file. Before creating a new versioned file (e.g. {city}_v4.html replacing {city}_v3.html), move the current version to Travel/archive/ first. Editing a file that already exists never triggers archiving — just edit it. Standing authorization — no asking needed.

What triggers auto-archiving:

  • A new guide version ships (e.g. before creating {city}_v4.html, move {city}_v3.html to Travel/archive/)
  • A photo is replaced or swapped out → old photo to Travel/archive/
  • Any photo or asset file from a previous guide version that does not carry forward into the new version → archive it. Every file not in active use in the new guide goes to Travel/archive/.
  • Any file superseded by a new versioned file → old version to Travel/archive/
  • Any file confirmed stale/unused during a sweep → to Travel/archive/

What does NOT trigger auto-archiving:

  • Editing a file in place — fixing content, correcting data, updating a rule, patching CSS, any change that keeps the same filename — just edit the file directly. Drive's revision history preserves the prior state. No archive step, no snapshot, no pre-edit copy.
  • Files that are current and in use — leave in place
  • Files in Brain/CORE RULES/ — user-only, never touched by Claude without explicit approval
  • An in-progress or incomplete guide build — any guide folder under Guides/ that has a _build/build_state.md (Phase 6 unchecked, or no guide HTML yet). Never archived, removed, deleted, or "cleaned up" — not even when it looks empty or stalled. A build tracker means the work is resumable: a stalled crib is finished by another crib from build_state.md, never discarded. There is no step, anywhere, that gets rid of incomplete guides. They are surfaced at session start by guide_tools.py start as a "maybe in progress — check later" flag, and that flag is the only action taken on them. Archiving one happens only on an explicit, per-case instruction to do so.

Archive destination: Travel/archive/ — locked. The single central archive for the entire Travel workspace. One folder, everything goes there. No per-guide, per-city, or per-folder archive/ (or Archive/) subfolders anywhere — not inside Guides/, Brain/, Brain/Reference/, or nested inside Travel/archive/ itself. Any crib that needs to archive moves the file into Travel/archive/ directly; it never creates a local archive folder. brain_check enforces this.

Wall 2 — Archive is access-locked. Claude does not access Travel/archive/ — for any purpose — without explicit per-session permission. This covers all operations: read, grep, copy, move, delete, or reference. Archive content is not read to learn rules or patterns — the current Brain is the source of truth; archives preserve history, not inform builds. Archive files are never moved back out of Travel/archive/ — unarchiving is not a Claude action. Restoring a file from archive is a manual operation — not a Claude action.

A guide build requires zero clarification. When asked to build a guide, all requirements are fully specified in the Brain files. Do not ask clarifying questions, do not use the questions UI, do not surface options, do not wait for confirmation before starting. Begin immediately and run the full build without stopping.

The Cowork AskUserQuestion tool is never invoked during any Travel task. Cowork's default behavior of asking clarifying questions before multi-step tasks is fully overridden by the Travel workspace rules. This applies to every Travel task without exception: guide builds, hotel research, trip updates, audits, shopping research, and any other task in this workspace. Start immediately, run end-to-end, no popup, no question menu, no waiting for a reply.

"Zero clarification" means zero user questions — not zero file reads. The CORE RULES files ARE the template and the format. Reading them is the build. A guide written without the Phase reads has no template — it will be wrong. Do not write a single line of HTML until the Phase 1 and Phase 2 reads are done.

All research actions during a build are pre-authorized.     • City name + number of days given → the day count in the request is the signal that the trip is not yet in Trips.html. Do not look for it. Run the hotel research playbook (Brain/Reference/Connectors.html Hotel Research section) to find the hotel. Use the stated day count. Continue building.
    Dates never appear in the guide (§ 7) and never block a build. Always use Day 1 / Day 2 / Day N numbering — never calendar dates in guide content. Never ask for dates.
  2. Read the mandatory CORE RULES files before writing any HTML. These are the template — not optional prereads:

    • Phase 2 (guide structure and content rules — before researching the city): Guide Structure.html · Stops Structure.html · Hotel Banner.html · Trip Overview.html · Brain/Reference/Toolbar.html · Brain/Reference/Navigation.html
    • Phase 3 (before locking any day): Day Structure.html
    • Phase 4 (per stop): Tickets.html · Motion Rule.html · Icon Order and Format.html
    • Phase 5 (each extra section): the matching *Extra Section*.html file — re-read at the start of that section
    Self-diagnostic: if any HTML has been written before Phase 2 reads are done, the build is already drifting. Stop, do the reads, start the HTML over.
  3. Build the entire guide in one pass following what the CORE RULES files say. No chunks, no returning later.
  4. When anything is unclear, make the best call the rules support and keep building — every gap Claude's tools can resolve is resolved in the same session. Only a question concerning a fix or edit to a Brain/CORE RULES/ file parks in ❓ Open Questions in the To Do List in Travel/To Do List/. Never block.
  5. Run the validator, fix all failures, log, then ship. Run python3 Brain/scripts/guide_tools.py validate <guide.html> — fix every ❌ failure, re-run, repeat until zero failures remain. Then append a validation entry with date and time to the audit log in Brain/Reference/: {City} {version} · validated {YYYY-MM-DD HH:MM} · ✅ {N} passed / 0 failed. Only then run the ship-gate: python3 Brain/scripts/guide_tools.py ship <guide.html>. Exit code 0 = done. A guide with open ❌ failures is not done — it is a fix-and-rerun loop, not a ship-gate call.

⚠️ STANDING ORDER — validator before any "done" claim (2026-05-30)
Any session that touches guide HTML — whether building from scratch, patching one section, fixing a single stop, or recovering from an API crash — must run validate_itinerary.py (or guide_tools.py validate) and reach 0 failures before telling the user the work is complete. The scope of the change is not an exemption. A patch session that fixes images, descriptions, or broken HTML is not done until the validator passes — the same gate as a full build. Saying "done," "fixed," "shipped," or any equivalent without a passing validator run in the current session is a violation of this rule. Check build_state.md Phase 6 on arrival; if it is unchecked, the guide is not shipped regardless of what a prior session wrote.

That is the bootstrap. Everything else lives in Guide Structure and the Docs it references — when those rules change, Guide Structure inherits and this file does not need to know.

Verified sources only

Every fact in the guide traces to one of three sources captured this build:

  • A brain rule.
  • The trip-doc row for this destination (hotel name and address only).
  • A live-verified research result.

No placeholders. Every row carries verified data. Placeholders, stub URLs, stub photos, and empty rows are fabrication.

Mid-build: only fix non-content infrastructure

Validator / CSS / template / script bugs are fixed during a build. Brain content rules (anything under Brain/CORE RULES/) stay untouched mid-build — content ambiguity parks in ❓ Open Questions in the To Do List; rule edits happen later, in a separate session.

Brain-first — bug-fix freeze during Brain repair sessions

When a session's goal is fixing Brain bugs (rule gaps, validator issues, CSS problems, CLAUDE.md updates), no guide rebuilds or PDF renders happen until the Brain work is done. The repair session closes with a clean Brain; guide output happens after. Mixing Brain repair with guide work creates a race: the guide may ship before the fix that was supposed to govern it is in place.

Root-cause pass — surface all bugs before fixing any

When asked to do a "root-cause pass" — or when a session uncovers multiple related bugs — inventory every instance before touching a single one. Fix order: broadest Brain fix first (rule or validator), then artifacts inherit. The trigger phrase is: "root-cause pass." Never fix one symptom while sibling symptoms wait.

Every fix lands in the Brain — never in a guide HTML

Bug fixes go to brain rules / CSS / validators — never to a one-off edit on a guide HTML file. Fix once in the Brain; every future guide inherits. A one-off guide edit leaves the Brain with the hole.

Validator scope — guide-shipping only

Validators check the guide HTML, build artifacts, and supporting scaffolding for ship integrity. They do NOT validate rule content (Rules for Claude, on-demand Docs, CORE RULES HTML files, etc.) — those don't ship; drift in them is not a validator concern.

The on-demand checks in validate_itinerary.py catch on-demand content LEAKING INTO a shipped guide (a 🌤 Weather heading inside guide HTML, delta.com URLs in stop blocks). That is guide-shipping correctness — within scope. Validating the on-demand Docs themselves is out of scope.

Validator check before any guide fix

When a violation is spotted in a guide — during a build, an audit, or any other work — write the validator check first. Fix the guide only after the check is in place and catching the violation. This applies even mid-build, even for a "quick fix." The guide fix without the validator check leaves the Brain with the hole.

Validator-first workflow — the locked five-phase sequence

"Work on the validator" means a locked five-phase sequence. Phases are never collapsed. The guide is never fixed before the validator catches everything. The point is to make the validator permanently smarter — not to patch the guide once.

  1. Phase 1 — AUDIT. Run the current validator + manually review the guide. Produce a complete list of every violation found — both what the validator already catches and what it misses. Do not touch the guide.
  2. Phase 2 — WRITE. For every violation the validator does not currently catch, write a new check. Do not touch the guide.
  3. Phase 3 — VERIFY. Run the validator. It must flag every violation found in Phase 1. When anything is missed, return to Phase 2. Do not touch the guide.
  4. Phase 4 — FIX. Only now fix the guide — all violations in one pass.
  5. Phase 5 — CONFIRM. Run the validator one final time. Guide must pass clean. Done.

Validator pre-check — search before writing any new check

Before any validator work — read the Change Cascade Map in Brain/Reference/ and work the Validator check cascade. It governs every downstream file that must follow.

Before writing any new check in validate_itinerary.py (Phase 2 of the locked workflow above):

  1. Search for the icon or concept. Grep the validator for the icon glyph, the rule keyword, and related phrases. Confirm no equivalent check already exists — a duplicate check adds noise and risks false positives.
  2. Find the correct guide-section block. validate_itinerary.py is organized in contiguous per-section blocks — one block per guide section: Getting Around, Day Trips by Train, Shows, Day Structure, Restaurants Near Hotel, Downtown Restaurants, Cappuccino, etc. Each block opens with a # ═══ header. Place any new check inside the matching section block. If no block yet exists for that section, create one using the # ═══ header pattern following existing conventions. A check placed outside its section block is scatter — the exact problem the per-section reorganization fixed — and breaks the file's enforcer structure.
  3. Update the Validator Index file (Validator Index.html) in Brain/Reference/. After adding any new check to validate_itinerary.py, add a matching entry to Validator Index.html in Brain/Reference/ under the correct section, with the appropriate ✅ / ❌ / ⚠️ status. This file is the living index of every check — a check added to the validator without a matching entry here is invisible to future review.

"Done" = every section verified

A guide is complete when:

  • Every stop carries a ticket box (🎟) or self-walk entry — with verified URLs and ratings.
  • Every stop carries a real photo or the explicit "No pictures found" negative-finding line.
  • Every restaurant entry carries address, hours, and walk time.
  • Every link is live-verified.
  • Every extra section is populated or carries its negative-finding line — except the conditional sections (❗️ Heads Up, Claude Inspiration, Skip List), which ship nothing when empty.
  • Every cross-link anchor resolves.

Partial builds surface explicitly. When a build cannot complete in one pass, the partial state is reported in the form: "Shipped X. Y is incomplete because Z."

The ship-gate is the arbiter. Exit code 0 means done. Any other exit code means not done.

Publishing guides

§ 6. DriftyCat — short warnings of things that keep breaking

One-line tripwires. The full rule for each lives in its CORE RULES HTML file — these are reminders for what Claude keeps drifting on.

  • ⚠️ Tour-first ALWAYS. Viator / GetYourGuide / TripAdvisor before skip-the-line, before venue site. Bar: 4.5+⭐ · ≥6 reviews. Full rule: Tours - Extra Section.html.
  • ⚠️ Zero money in shipped guides. No $, €, £, ¥, no ISO codes — ever. USD is not an exception.
  • ⚠️ Tilde has one home — ⏰. The tilde ~ is valid only as the ⏰ Avg Time Spent prefix (an estimate, e.g. ⏰ ~1 hr). It is banned everywhere else — never on ⏳ Duration (exact: ⏳ 45 min), never on walk/ride/start/travel-time values, never in descriptions or addresses. One carve-out, no others. Full rule: Icon Order and Format.html.
  • ⚠️ No placeholders. {TBD} / {TODO} / "fill in later" / "tour TBD" = fabrication.
  • ⚠️ No fabrication. Every fact live-verified this build. Confirmed sources are the venue's own site, Google Maps, or a verifiable third-party listing. "Probably" / "usually" / memory from past builds are NOT sources. A guess dressed as a fact is worse than a blank.
  • ⚠️ No hedging language in factual rows. Every factual row (🏛 hours, 🚶 walk time, 🚕 ride time, ⏰ duration, 📍 address, 🎟 booking link) must come from a confirmed source. Hedging language is the fingerprint of invented or unverified data. The following phrases are banned in factual rows — the validator hard-fails on them:
    "typically open" · "usually open" · "generally open" · "hours may vary" · "hours vary" · "subject to change" · "confirm hours" · "confirm times" · "check schedule" · "check the website for hours" · "see website for times" · "varies by season" · "seasonal hours" · "prices vary" · "cost varies" · "typically takes" · "usually takes" · "approximately N" · "around N" · "estimated N" · "may be" / "might be" / "could be" (when N or the phrase appears in a factual data point).
    When a value cannot be confirmed — look it up. When it can't be confirmed — omit the row. Hedges are invisible inventory: they look like data, but they're guesses.
    Carve-out — ⏰~ only: the ⏰~ Avg time spent row is exempt from this hedging ban, because it is an inherent estimate already marked as such with the ~ prefix. The exemption applies to the ⏰~ row and nothing else — ⏳ and every other factual row (🏛 hours, 🚶 walk time, 🚕 ride time, 📍 address, 🎟 booking link) remain fully under the ban.
  • ⚠️ Every link live-verified — every time. Including every edit inside a session. A one-character URL change re-runs verification. Bot-blocked platforms (Viator, GetYourGuide, TheFork, Michelin, TripAdvisor) verify via site: search-result inspection, not direct fetch.

§ 9. Audit

Triggered only by the word "audit" (alone or in a phrase). Workflow:

  1. Thorough review of files under Travel/ to identify what is wrong, outdated, or broken.
  2. Identify what rule / validator / CSS would catch each finding — strengthen those, not the artifact.
  3. Append findings to the audit log in Brain/Reference/.

Audit is additive — fixes go in, removals get parked for permission. Run proactively when the last audit is more than 7 days old.

Polish vs. removal. Polish = clearer phrasing within a paragraph, tightening sentence-level. Consolidating or removing rules is NOT polish — those park for explicit permission. Past audits have stripped rules under the banner of "tightening" — that is the failure mode this rule catches.

"List fixes" without the word "audit" = surface what is already known broken. Not a 27-file crawl. The two processes have different costs and different outputs.

§ 10. Session close-out checklist

Before ending any non-trivial work session, run through these five items:

  1. Brain-check. Run python3 Brain/scripts/brain_check.py — zero warnings = clean. Any flag = fix before leaving.
  2. Audit log. Append a session entry to the audit log in Brain/Reference/ — date, what was done, what was found, what was fixed.
  3. Versioned files only. Any guide HTML that shipped a new version this session? Confirm the old version was moved to Travel/archive/ before the new one was created. In-place edits (same filename) need no archive — Drive revision history covers them.
  4. Decisions log (Brain.md Part 2). Any non-trivial judgment calls made this session that are not in the decisions log (Part 2 of Brain/Reference/Brain.md)? Append them now.
  5. Developer files. Six files in Brain/Reference/ must stay current — update them before the session ends. No asking permission. Stale = drift.
    • the Validator Index file (Validator Index.html) in Brain/Reference/ — add every new check added to validate_itinerary.py or brain_check.py this session.
    • the Guide Entry Counts file (Guide Entry Counts.html) in Brain/Reference/ — update any count, minimum, or threshold that changed this session (section minimums, char caps, walk caps, etc.).
    • the Rule Dependencies file (Rule Dependencies.html) in Brain/Reference/ — update any icon, threshold, or shared concept that moved, was renamed, or changed scope this session.
    • the Ship Checklist file (Ship Checklist.html) in Brain/Reference/ — update it whenever a section is added or removed from the canonical extras order, and whenever a new validator script is added.
    • the Emoji Library in Brain/Reference/ — update whenever a new emoji is approved for use or an existing one is locked out or reclassified.
    • the Connectors file in Brain/Reference/ — update whenever a connector is added, removed, tested live for the first time, or its workflow changes.
    These are maintained by Claude. If a session ends with new checks, new counts, or changed rules — and these files were not updated — that session left drift behind.

A session that ends without these five steps leaves invisible debt: Brain drift goes undetected, judgment calls evaporate, and the next session starts with stale context.

§ 11. Google Calendar management

Hotel banners

Each hotel stay has one all-day banner event spanning check-in day through check-out day (inclusive). The Google Calendar API uses an exclusive end date for all-day events — the end date is set to check-out day + 1. Banner content: hotel name and address only.

Flight events

Flight events carry flight number, route, departure time, and arrival time only. No booking references, seat numbers, or other details.

Color coding

Events are color-coded by participant. Grape (color ID 3) = primary traveler's solo events. Sage (color ID 2) = companion's solo events. Banana (color ID 5) = events shared by both. Events involving only one traveler are created separately; shared events are one event.

Activity and tour events

Activity and tour event content: date, start time, meeting point, and cash required (if any). No other information.

§ 12. Format-exception files

Five CORE RULES files are Claude reference files, not guide-builder rule documents. They describe technical formats for Claude's own use — JSON structures, URL patterns, file paths, HTML snippets, illustrative bad patterns, or required-reads lists that use controlled vocabulary. Standard formatting rules do not apply to them in full.

The five files are: Links.html, Photos Rules.html, Rules for Claude.html, Guide Structure.html, Claude Inspiration - Extra Section.html.

What is different in these files: curly-brace notation ({like this}) is intentional technical notation, not a template placeholder. <em> tags mark illustrative phrases. Validator checks that would flag these patterns are skipped for these files. Each file carries a "Format exception" notice below its read-only banner stating what is intentional.

The validator enforces this via the FORMAT_EXCEPTION_FILES constant in doc_workshop_validator.py. Any future check that does not apply to these files gates itself with if path.name not in FORMAT_EXCEPTION_FILES.
